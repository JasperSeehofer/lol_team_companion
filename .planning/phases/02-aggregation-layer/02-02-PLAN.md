   Add `pub async fn migrate_champion_names(db: &Surreal<Db>) -> DbResult<()>`:
   - Call `data_dragon::fetch_champions().await` to get canonical champion list. If fetch fails, log warning and return Ok (migration is best-effort on startup).
   - Build a `HashMap<String, String>` for fuzzy lookup: for each champion, insert `lowercase_stripped(c.id) -> c.id` and `lowercase_stripped(c.name) -> c.id` where `lowercase_stripped` removes non-alphanumeric chars and lowercases.

   Add `pub async fn get_dashboard_summary(db: &Surreal<Db>, team_id: &str) -> DbResult<DashboardSummary>`:
   - Strip `team:` prefix from team_id for `type::record` usage.
   - Use a 5-statement batched query (see RESEARCH.md Example 2):
     - Statement 0: Top 3 open/in_progress action items (SELECT id, text, created_at ORDER BY created_at DESC LIMIT 3)
     - Statement 1: Total open/in_progress action item count (SELECT count() GROUP ALL)
     - Statement 2: Last 5 post-game learnings with improvements (SELECT id, improvements, created_at ORDER BY created_at DESC LIMIT 5)
     - Statement 3: Count of drafts without game plans
     - Statement 4: Count of game plans without post-game reviews
   - For action items SELECT, include `created_at` in the SELECT clause since ORDER BY requires it (CLAUDE.md rule 40).
   - Use `unwrap_or_default()` for all `.take()` calls.
   - For count queries, define an inline `#[derive(Debug, Deserialize, SurrealValue)] struct CountResult { n: i64 }` and take the first result's `.n` or default to 0.
   - Build `ActionItemPreview` and `PostGamePreview` from the Db results.

3. **Pool gap computation**:
   Add `pub fn compute_pool_gaps(pool_entries: &[ChampionPoolEntry], champions: &[Champion], opponent_champions: &[String]) -> Vec<PoolGapWarning>` (non-async, pure Rust):
   - Group pool entries by `(user_id, role)`.
   - For each group, look up class tags for each champion via a `HashMap<String, Vec<String>>` built from `champions`.
   - Count class occurrences. If total > 0 and any single class has >= 70% of entries, flag as dominant.
   - Check all 6 classes (Fighter, Mage, Assassin, Tank, Marksman, Support). If a class has 0 coverage AND that class appears in `opponent_champions`' tags, flag as `opponent_escalated: true`.
   - Collect warnings per user per role.
   - Call this from `get_dashboard_summary` after fetching pool entries for all team members (using the team member -> user ID -> pool entry join pattern from RESEARCH.md Example 1) and fetching opponent data.

  <name>Task 2: Champion performance summary queries</name>
  <files>src/server/db.rs</files>
  <behavior>
    - get_champion_performance_summary(db, team_id, user_id, window_days, min_games) returns per-champion stats for a specific player
    - get_team_champion_performance(db, team_id, window_days, min_games) returns team-wide champion stats
    - Default window: last 30 days OR last 20 games (whichever is more) per user decision
    - Both return Ok(Vec::new()) when no data exists
    - Stats aggregate from: draft picks (games_in_draft), match stats (games_in_match, wins_in_match), game plans (games_in_plan), post-game reviews (post_game_wins, post_game_losses)
  </behavior>
  <action>
1. **Per-player champion performance** in `db.rs`:
   Add `pub async fn get_champion_performance_summary(db: &Surreal<Db>, team_id: &str, user_id: &str, window_days: Option<i64>, min_games: Option<usize>) -> DbResult<Vec<ChampionPerformanceSummary>>`:
   - **Time window logic (per user decision: "last 30 days OR last 20 games, whichever is more"):**
     1. Default `window_days` to 30 and `min_games` to 20.
     2. Calculate the day-based cutoff: `chrono::Utc::now() - chrono::Duration::days(window_days)`.
     3. First, query match stats count within the day-based window: `SELECT count() FROM player_match_stats WHERE user = type::record('user', $user_key) AND created_at >= $cutoff GROUP ALL`.
     4. If the count is less than `min_games`, extend the window: query the Nth oldest game's timestamp (`SELECT created_at FROM player_match_stats WHERE user = type::record('user', $user_key) ORDER BY created_at DESC LIMIT 1 START $min_games - 1`) and use that as the new cutoff. If fewer than `min_games` total games exist, use the earliest game's timestamp (effectively no time cutoff).
     5. Use the resolved cutoff for all subsequent queries.
   - Batched query with 4 statements (using the resolved cutoff):
     - Draft actions where the draft belongs to the team and champion was picked (not banned) by this user's side, within window. NOTE: drafts don't have per-user attribution, so count all team-side picks.
     - Match stats (`player_match_stats`) for this user within window: champion, win.
     - Game plan champions (`our_champions` array) for team within window.
     - Post-game learnings for team within window (win/loss from the linked game plan or match outcome).
   - Aggregate in Rust using `HashMap<String, ChampionPerformanceSummary>`:
     - For each draft pick: increment `games_in_draft`.
     - For each match stat: increment `games_in_match`, conditionally `wins_in_match`.
     - For each game plan champion: increment `games_in_plan`.
     - For post-game: use the review's outcome to determine win/loss attribution per champion.
   - Return sorted by `games_in_match` descending.
   - If user has no team or no data, return `Ok(Vec::new())`.

2. **Team-wide champion performance** in `db.rs`:
   Add `pub async fn get_team_champion_performance(db: &Surreal<Db>, team_id: &str, window_days: Option<i64>, min_games: Option<usize>) -> DbResult<Vec<ChampionPerformanceSummary>>`:
   - Same logic as per-player but aggregates across all team members.
   - For the "30 days OR 20 games" rule, apply it team-wide: count all team member matches within 30 days. If total < 20, extend window to include at least 20 games across all team members.
   - Get all team member user IDs first, then query match stats using `user IN [list]` pattern.
   - Draft picks: all team-side picks (no user filter needed since drafts are already team-scoped).
   - Return sorted by `games_in_match` descending.

3. Both functions must handle the "no team" case by returning `Ok(Vec::new())` early (CLAUDE.md rule 44).

4. Add `#[cfg(test)]` unit tests:

   - Extract the Rust-side aggregation logic into a testable helper function `aggregate_champion_performance(drafts, matches, plans, post_games) -> Vec<ChampionPerformanceSummary>` so it can be tested without a live DB.
   - `test_aggregate_champion_performance`: feed known inputs, verify correct counts
