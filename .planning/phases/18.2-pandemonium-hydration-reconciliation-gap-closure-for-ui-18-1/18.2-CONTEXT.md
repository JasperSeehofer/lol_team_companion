# Phase 18.2: Pandemonium Hydration Reconciliation (gap closure for UI-18.1-HYDRATE-01) — Context

**Gathered:** 2026-05-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Restore client-side interactivity on every Pandemonium-themed page by eliminating the SSR/hydrate divergence in a Pandemonium structural primitive (or set of primitives) that causes `tachys-0.2.14/src/html/mod.rs:217 called Option::unwrap() on a None value` during WASM hydration. SSR HTML continues to render correctly post-fix (Phase 18.1 proved that); the WASM client successfully reconciles against it and attaches event handlers (mode toggle, theme toggle, drag-drop) on Pandemonium pages. A new e2e regression spec catches both the panic AND inert-after-SSR failure shapes going forward.

**In scope:**
- Diagnosing which Pandemonium structural primitive(s) emit SSR markup that diverges from the WASM render of the same component.
- Fixing the divergence (or in the bounded fallback case, emitting a hydration-compatible SSR placeholder).
- Adding `e2e/tests/hydration-no-panic.spec.ts` as the regression gate.
- Writing `18.2-DIAGNOSIS.md` as the audit trail + reusable playbook.
- Re-running the Phase 18.1-03 agent-browser sweep with WASM event handlers exercised.

**Out of scope:**
- Any change to Phase 18's region API (`region: String` prop convention stays).
- Phase 18.1's SSR injection layer (works correctly; treat as foundation).
- Mode-toggle wiring, ModeToggle component, DB persistence (all Phase 18-08).
- Bug-Report Widget (Phase 19).
- Demacia branch redesign — only Demacia parity (no regression) is required.
- Re-baselining 18-09 visual baselines (already done in Phase 18.1-02).

## Root cause recap

Phase 18.1's SSR-theme-injection middleware made the Pandemonium structural branches reachable at runtime for the first time. The 18.1-03 agent-browser sweep (2026-05-25) found that every Pandemonium-themed page (`/draft`, `/solo`, `/team/dashboard`, `/tree-drafter`, `/champion-pool`, `/match/:id`, `/post-game`, plus shared sub-views) panics during WASM hydration:

```
[error] panicked at /home/jasper/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/tachys-0.2.14/src/html/mod.rs:217:14:
called `Option::unwrap()` on a `None` value
```

Demacia pages hydrate cleanly. The panic kills all WASM event handlers on Pandemonium pages — mode toggle, theme toggle, drag-drop, and other interactivity are silently dead. Pre-18.1 this was hidden because SSR always rendered the Demacia branch; both sides agreed.

Recorded as **UI-18.1-HYDRATE-01** (severity: bug) in `.planning/phases/18-region-variants/18-HUMAN-UAT.md`. Evidence in `.planning/phases/18.1-ssr-theme-injection-gap-closure-for-ui-18-runtime-01/18.1-03-EVIDENCE.md` (Investigation Flag 1).

## Why this phase exists before Phase 19

Phase 19 (Bug-Report Widget) mounts a floating click-to-report element on every authenticated page that needs WASM event handlers. If 18.2 isn't fixed first, the bug-report widget is inert under Pandemonium — the widget can't report its own broken hydration. Pandemonium UX is also incomplete (visible-but-inert) which is worse than not-shipped.

<decisions>
## Implementation Decisions

### Bisection Strategy
- **D-01 — Method:** Short-circuit-and-reload. Per Pandemonium primitive, temporarily return the Demacia branch (or empty view) from the `if is_pandemonium { ... }` arm; reload a representative Pandemonium page; observe the JS console for the `tachys` panic. The component whose neutralization eliminates the panic is the culprit (or one of several). Cheap, no instrumentation, agent-browser-driven.
- **D-02 — Bisection order:** Leaves first. Start with the smallest leaf primitives (`RiotTape`, `ToxicSticker`, `WaxSeal`, `Glitch`, halftone/oil-spill backgrounds) before the per-page Pandemonium rebuilds (DraftLedgerView Pandemonium arm, the 7-section team-dashboard Pandemonium rebuild, SoloForge, SoloJournal, TeamGameDayBriefView). Leaves are likeliest culprits because they're used many times per page and have the most opportunity for an SSR/WASM divergence (random hashes, conditional class strings, conditional `{move ||}` blocks).
- **D-03 — Documentation:** Inline diagnosis file `18.2-DIAGNOSIS.md` in the phase directory. Records which primitives were short-circuited in which order, the observed console state at each step, the culprit's identity, the root cause, why the chosen fix works, and (if applicable) the deferred-real-fix entry.
- **D-04 — Time-box / escalation:** If short-circuit-and-reload doesn't isolate the culprit in ~1 day (single Wave-1 plan), escalate by upgrading to WASM-side render-tracing (add `web_sys::console::log` calls at the top of each region primitive's render fn so render order/IDs print on hydrate; walk the tree to find the divergence point). Escalation is documented in `18.2-DIAGNOSIS.md` and does not require user check-in.

### Regression-Gate Scope
- **D-05 — Spec assertions:** New e2e spec asserts BOTH failure shapes. (1) `page.on('pageerror')` + console-error listener fails on any text matching `/panicked at|Option::unwrap.*None|tachys/i`. (2) Click each scoped page's mode-toggle / theme-toggle / draft drag-drop and assert observable DOM change (see Interactivity Assertion List below). Generic `console.error` (404 image, deprecation warning, etc.) does NOT fail — narrow signal for hydration failure.
- **D-06 — Spec location:** New dedicated spec `e2e/tests/hydration-no-panic.spec.ts`. Discoverable by name; becomes the canonical "hydration health" gate that future region/theme phases also exercise. Not folded into `region-diff.spec.ts` (different concern: pixel diff vs. hydration integrity) and not added to `fixtures.ts` as a global default (avoids slowing every test with the pageerror listener overhead).
- **D-07 — Region coverage:** Both regions. The gate runs the panic + interactivity sweep against both Demacia AND Pandemonium per scoped page. Guards against the symmetric bug (a future change that breaks Demacia hydration) AND catches if the 18.2 fix accidentally regresses Demacia. Demacia parity is verified by the same spec — no separate Demacia smoke test needed.
- **D-08 — Error-level policy:** Fail on `page.on('pageerror')` events AND on console messages matching `/panicked at|Option::unwrap.*None|tachys/i`. Not "all console errors fail" (false-positive prone) and not "page errors only" (Leptos panics also surface as console.error in some hydrate codepaths).

### Fix-vs-Fallback Policy
- **D-09 — Preference:** Real Leptos-side fix preferred. Reshape the divergent component so both render paths produce identical markup (e.g. wrap conditional in `{move || }`, add `Show fallback=...`, normalize node count between branches, eliminate non-deterministic class-string ordering). The codebase stays honest; future hydration phases don't trip over cfg(ssr) debt.
- **D-10 — Fallback gate:** If the real fix requires reshaping more than one primitive's render signature OR is estimated > 1 day of work (after diagnosis), fall back to a `cfg(feature = "ssr")` placeholder that emits the structural skeleton WASM expects (same anchors, same node count). The cost gate is enforced in plan-phase based on the diagnosis findings. NOT a hair-trigger fallback (`> 4h`) — phase has time budget for a real fix when feasible.
- **D-11 — Fallback tracking:** Three-place audit trail when fallback fires: (a) v1.4 roadmap backlog entry citing the affected primitive and the structural change required; (b) inline `// HYDRATE-FALLBACK: see .planning/phases/18.2/18.2-DIAGNOSIS.md` comment at the cfg(ssr) site; (c) `18.2-DIAGNOSIS.md` adds a "Deferred real fix" section explaining what the fallback skips and what a future refactor must do.
- **D-12 — Visual parity under fallback:** SSR placeholder emits the structural skeleton (anchors, node count) WASM expects. The WASM-side render still produces the full Pandemonium primitive content on hydrate. User sees Pandemonium as intended; the SSR markup just doesn't pre-render the rich variant content (acceptable — it lands within milliseconds of hydration). Degrading to the Demacia branch under Pandemonium is NOT acceptable — that breaks Phase 18's region-variants premise.

### Interactivity Assertion List
- **D-13 — Core triad:** The minimum WASM-handler assertion set the regression spec must verify on Pandemonium (and symmetrically on Demacia):
  1. **Mode-toggle click** on `/draft`, `/solo`, `/team/dashboard` under Pandemonium changes the visible mode signal (observable DOM marker change).
  2. **Theme-toggle click** on a Pandemonium page reaches Demacia and back without panic and without losing client state.
  3. **Draft slot drag-drop** on `/draft` fires `on:dragstart` and updates the slot (exercises the heavier DataTransfer API path documented at `src/components/draft_board.rs:91-105, 207-217`).
- **D-14 — Observable-change assertion shape:** After clicking each interaction trigger, poll for the DOM marker of the new state to appear within 2s. If not, fail with a specific error message (e.g. "mode-toggle silently inert under Pandemonium on /draft"). Catches inert-after-SSR; gives useful failure messages. NOT just "before/after DOM differs" (coarse) and NOT "compiled-WASM event-listener inspection" (brittle across Leptos versions).
- **D-15 — Persistence scope:** No DB-write + reload + state-restored assertions in this spec. Persistence is already covered by Phase 18-08's mode-persist tests and Phase 18.1's cookie tests. Phase 18.2's gate scopes narrowly to: WASM handler attached + visible-state changes. Keeps the spec fast and focused on the hydration concern.
- **D-16 — Demacia parity assertion:** Same spec runs both regions (re-uses D-07). If a 18.2 change accidentally breaks Demacia mode-toggle, the spec fails. No separate Demacia smoke needed.

### Claude's Discretion
- **Plan-wave structure:** Likely 2 waves — Wave 1: bisect + fix (or fallback) + diagnosis file; Wave 2: hydration-no-panic.spec.ts + agent-browser sweep + flip UI-18.1-HYDRATE-01 → RESOLVED in 18-HUMAN-UAT.md. Plan-phase decides whether to split fix and gate into separate plans or bundle.
- **REQ-7 re-verification:** Utility-route region-agnosticism was confirmed in Phase 18.1-03 (3/3 PASS). Plan-phase decides whether to re-run that spot-check (low cost: ~5min agent-browser) or trust Phase 18.1's evidence (no per-utility-route Pandemonium primitives changed in 18.2, so structurally there's no path to regress REQ-7).
- **WASM tracing logging scope on escalation:** If D-04 escalation triggers, plan-phase decides whether to instrument all region primitives at once or progressively. Trade-off: faster signal vs. console noise.
- **Multiple culprits handling:** If bisection finds >1 culprit primitive, plan-phase decides whether to fix all in one plan or sequence. Bundling makes sense if they share a root cause (e.g. all use the same non-deterministic class-string pattern).

### Established Patterns
- **Region prop sourcing:** Each scoped page does `let region = use_context::<InitialTheme>().0` once at entry and passes `region: String` down. The fix must not change this convention (Phase 18 + 18.1 + downstream pages all depend on it).
- **WASM safety rule:** `.claude/rules/wasm-patterns.md` rule 56 — never use `.unwrap()` in event handlers. The current bug is upstream of this rule (tachys's own unwrap, not user code), but the rule's spirit applies: any 18.2 fix must not introduce new panics.
- **Hydration-friendly conditionals:** Leptos 0.8 convention — conditional content that differs SSR vs WASM must go inside `{move || ...}` or `Show fallback=...`. Static-at-SSR conditionals (`if region == "pandemonium" { ... } else { ... }` at the top of a component) work IFF both branches return the same node-count shape after `.into_any()`. Likely-culprit class.
- **Atomic-commit per bisection step:** Each short-circuit + reload + observe cycle becomes a commit on a scratch branch (or stash + reapply); the final fix commit follows the diagnosis. Keeps the bisect trail in git log even though the canonical record is `18.2-DIAGNOSIS.md`.

## Specific Ideas

- **Concrete repro:** Open `/solo` with `Cookie: lol_companion_theme=pandemonium`, observe the console for the tachys mod.rs:217 panic. Demacia equivalent (cookie=demacia) produces no panic. (Same repro path used by Phase 18.1-03 sweep.)
- **Likeliest-culprit hunch from 18.1-03 evidence:** The bug is universal across all 7 scoped routes, so the culprit is shared — either a leaf primitive used by every Pandemonium page (RiotTape, halftone background, Glitch) or a shared shell component (region/layout's Card). The page-level views are too varied to all share the same divergence pattern unless they all wrap a common primitive.
- **Suspense-timing caveat to respect:** Phase 18.1-03 noted a Leptos SSR Suspense race on cold first fetch of `/profile` Pandemonium. The bisect must warm Suspense resources before observing the panic (or it'll false-attribute the bug to Suspense). Warm-fetch the page once, then capture the second-fetch console.
- **Mode-toggle SSR-default note:** Per Phase 18-08 + 18.1-03 Flag 2, mode-toggle SSR-renders the region-coupled DEFAULT mode (not the user's DB-persisted mode). The interactivity assertion clicks the mode-toggle to change AWAY from the SSR default, so the assertion is independent of the SSR-default question.
