### Why the panic STILL reproduces despite leaf neutralization

The `tachys-0.2.14/src/html/mod.rs:217` site is `InertElement::hydrate` per RESEARCH.md "Tachys panic site (verified from source on disk)" — it fires when the WASM hydrate walk reaches an element whose owner-document is `None`, which happens when the SSR HTML's node count or ordering does NOT match what WASM expects, causing the cursor to walk OFF the end of the materialized DOM. Short-circuiting a leaf primitive REMOVES its markup from SSR but also removes its expected slot from WASM's render plan — IF the leaf were the desync source, both sides would shrink symmetrically and the cursor would no longer walk off. Since the panic persists, the desync source is in markup that is STILL rendered after the short-circuit — i.e., it's in the surviving parents/siblings/peers, not in the neutralized leaf.

**CRITICAL FINDING — culprit identified:** Short-circuiting CompanionSigil's Pandemonium arm to `view! { <></> }` did NOT clear the panic, BUT the panic SHAPE shifted from `html/mod.rs:217 InertElement::hydrate Option::unwrap()` to `hydration.rs:184 "Unrecoverable hydration error"` with an EXPLICIT human-readable message pinpointing `src/components/region/ornaments.rs:156:14` (the OUTER `<div class="flex items-center gap-2.5">` of CompanionSigil's **Demacia** arm). The error message: *"The framework expected an HTML &lt;div&gt; element, but found this instead: nodeType: 8 (comment node)"*.

**Interpretation:** Under the mutation, SSR (which sees the Pandemonium-cookie request, has `InitialTheme("pandemonium")` in context, computes `is_pandemonium=true`) emits `view! { <></> }` — a Leptos Fragment which renders to a comment-node placeholder. WASM hydrate, however, EXPECTED a `<div>` at that position. The expected `<div>` corresponds to the **Demacia arm** of CompanionSigil (`ornaments.rs:156`). This means **the WASM hydrate path is evaluating `is_pandemonium = FALSE` even though the page was loaded under the Pandemonium cookie.**

**Root cause hypothesis (high confidence):** CompanionSigil reads its branch from `use_context::<InitialTheme>().map(|t| t.0 == "pandemonium").unwrap_or(false)` (`ornaments.rs:137-139`). SSR provides `InitialTheme` via `main.rs:96`'s `provide_context(theme)` inside `leptos_routes_with_context`. The WASM hydrate path has **no equivalent `provide_context` call** — `App::App()` in `src/app.rs:79` does not re-provide `InitialTheme`. Therefore on WASM hydrate, `use_context::<InitialTheme>()` returns `None`, the `.unwrap_or(false)` triggers, `is_pandemonium = false`, and the Demacia arm is rendered. Under the cookie=pandemonium request, this causes a structural mismatch (SSR emits Pandemonium arm; WASM hydrates expecting Demacia arm) — the original `html/mod.rs:217 Option::unwrap()` panic IS this mismatch detected at a later cursor walk site.

**Note: this hypothesis explains the original panic shape too.** In the un-mutated CompanionSigil, SSR emits `<span class="font-glitch">"COMPANION_"</span>` (Pandemonium arm) and WASM expects `<div class="flex">...<svg/>...<div>"COMPANION"</div></div>` (Demacia arm). The cursor walks the span, hits surprise children/structure, walks off the end → `Option::unwrap()` on None at `html/mod.rs:217`. Under the short-circuit mutation, SSR emits a comment marker and WASM expects `<div>` — Leptos detects this as a clean "type mismatch" hydration error (the new explicit error path) rather than walking off and unwrap-on-None'ing.

**Conclusion: CompanionSigil IS the (or a) culprit.** The smoking gun is the explicit error message pointing to its Demacia arm `<div>`. Revert applied; cycles 6-8 still run to check for secondary culprits per Pitfall 4. Plan 03's fix pattern is now clear: either (a) re-provide `InitialTheme` on the WASM hydrate side via `App::App()`, or (b) refactor CompanionSigil to take `region: String` as a prop like every other region primitive (CompanionSigil's docstring already flags itself as the legacy outlier).
