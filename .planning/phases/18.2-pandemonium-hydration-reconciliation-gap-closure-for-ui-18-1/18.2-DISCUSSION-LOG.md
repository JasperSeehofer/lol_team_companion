## Bisection Strategy

### Q1: How should we hunt the culprit primitive?

| Option | Description | Selected |
|--------|-------------|----------|
| Short-circuit + reload | Per Pandemonium primitive, temporarily return the Demacia branch (or empty view) from the `if is_pandemonium` arm; reload a representative Pandemonium page; observe console. The component whose neutralization eliminates the panic is the culprit. Cheap, no instrumentation, agent-browser-driven. | ✓ |
| WASM-side render tracing | Add `web_sys::console::log` calls at the top of each region primitive's render fn so the order/IDs print on hydrate. Walk the tree to find the divergence point. More information, but more invasive code churn and needs cleanup after. | |
| Page-level bisect first, primitive second | First narrow to which scoped page (or shared view) reliably panics with the smallest set of primitives. Then short-circuit primitives within that page. Two-level hunt, slower but more deterministic with overlapping primitives. | |

**User's choice:** Short-circuit + reload (Recommended)
**Notes:** Cheapest signal; the diagnosis file (D-03) preserves the bisect trail.

### Q2: Should the bisection start from the smallest leaf primitives or from the page-level views?

| Option | Description | Selected |
|--------|-------------|----------|
| Leaves first | Start with RiotTape, ToxicSticker, WaxSeal, Glitch, halftone/oil-spill backgrounds. Most likely culprits because they're used many times per page and have the most opportunity for an SSR/WASM divergence. | ✓ |
| Pages first | Start with the per-page Pandemonium rebuilds. Faster early signal because each page = many primitives at once, but harder to localize. | |
| Smallest panicking page first | Find the smallest Pandemonium page that panics, then bisect within it. Combines specificity with efficiency. | |

**User's choice:** Leaves first (Recommended)
**Notes:** Hunch from 18.1-03 evidence: universal panic across 7 routes → shared primitive most likely.

### Q3: How should the bisection work be documented?

| Option | Description | Selected |
|--------|-------------|----------|
| Inline diagnosis file (`18.2-DIAGNOSIS.md`) | Records short-circuit order, observed console state, culprit identity, root cause, and why the chosen fix works. Becomes audit trail + reusable playbook for future hydration bugs. | ✓ |
| Commit-log only | Each bisection step is its own commit with a descriptive message; the diagnosis story lives in `git log`. | |
| Both | Atomic commits per bisection step AND a summary file. Most thorough; small redundancy. | |

**User's choice:** Inline diagnosis file (Recommended)
**Notes:** Single canonical record; commits remain as supporting evidence in git log.

### Q4: What's the bail-out plan if bisection takes longer than expected?

| Option | Description | Selected |
|--------|-------------|----------|
| Time-box to 1 day, then escalate | If short-circuit-and-reload doesn't isolate the culprit in ~1 day (single Wave-1 plan), escalate by upgrading to WASM-side render tracing before continuing. Documented in the diagnosis file. | ✓ |
| No time-box — finish the bisect | Take however long it takes. Phase 19 doesn't ship anyway until Pandemonium hydrates. | |
| Open a `/gsd:debug` session if stuck >4h | Switch into the GSD debug workflow (persistent state across context resets) if the bisect isn't yielding signal within ~4 hours. | |

**User's choice:** Time-box to 1 day, then escalate (Recommended)
**Notes:** Escalation is in-phase (no user check-in needed), documented in diagnosis file.

## Regression-Gate Scope

### Q5: What should the new e2e regression spec assert?

| Option | Description | Selected |
|--------|-------------|----------|
| Panic + interactivity | `pageerror`/console-error listener fails on `Option::unwrap()`/`tachys` text; click mode-toggle on each scoped Pandemonium page and assert visible-mode signal changed. Catches both panic AND inert-after-SSR. | ✓ |
| Panic-only (minimal) | Only assert no hydration panic. Misses inert-after-SSR. | |
| Panic + interactivity + accessibility | Add aria-role / focus management checks. Broader gate, more maintenance surface. | |

**User's choice:** Panic + interactivity (Recommended)
**Notes:** Two failure shapes to catch; accessibility is its own future phase.

### Q6: Where should the regression spec live?

| Option | Description | Selected |
|--------|-------------|----------|
| New spec: `hydration-no-panic.spec.ts` | Dedicated spec, discoverable by name. Becomes canonical "hydration health" gate that future region/theme phases also exercise. | ✓ |
| Extend `region-diff.spec.ts` | Add to the existing 18-09 region-diff spec. Tighter coupling, less file sprawl. | |
| Add to `fixtures.ts` as shared assertion | Every authed Playwright test runs the pageerror listener. Catches the bug everywhere; risks slowing all tests. | |

**User's choice:** New spec (Recommended)
**Notes:** Distinct concern from pixel-diff; gate is reusable for future region/theme work.

### Q7: Should the gate run under both regions, or Pandemonium-only?

| Option | Description | Selected |
|--------|-------------|----------|
| Both regions | Panic + interactivity sweep against Demacia AND Pandemonium per scoped page. Guards against symmetric bug; catches if 18.2 fix accidentally regresses Demacia. | ✓ |
| Pandemonium-only | Pandemonium pages only; Demacia implicitly verified by existing tests. Faster. Misses symmetric-regression case. | |

**User's choice:** Both regions (Recommended)
**Notes:** Demacia parity (D-16) reuses the same spec; no separate Demacia smoke needed.

### Q8: Should the gate fail the build on console.error too, or only on actual page panics?

| Option | Description | Selected |
|--------|-------------|----------|
| Page errors + tachys/unwrap text in console | Fail on `pageerror` AND console messages matching `/panicked at\|Option::unwrap.*None\|tachys/i`. Generic console.error does NOT fail. | ✓ |
| All console errors fail | Strictest; high false-positive rate. | |
| Page errors only | Only `pageerror`. Narrowest; may miss Leptos panics that only surface as console.error in some hydrate codepaths. | |

**User's choice:** Page errors + tachys/unwrap text in console (Recommended)
**Notes:** Narrow signal; avoids false positives from third-party warnings.

## Fix-vs-Fallback Policy

### Q9: If the divergent component is structurally hard to refactor, is an SSR-side placeholder acceptable as a fallback?

| Option | Description | Selected |
|--------|-------------|----------|
| Real fix preferred; fallback allowed with diagnosis | Try the real fix first. If refactor cost > 1 plan, fall back to `cfg(ssr)` placeholder + recorded follow-up. Pragmatic; explicit cost gate. | ✓ |
| Hard 'real fix only' | No SSR placeholder fallback. May stall the phase. | |
| Whichever is faster, no policy | Decide on the spot. Risks cfg(ssr) debt. | |

**User's choice:** Real fix preferred; fallback allowed with diagnosis (Recommended)
**Notes:** Pragmatic gate with auditable cost threshold.

### Q10: What's the cost gate that triggers the fallback?

| Option | Description | Selected |
|--------|-------------|----------|
| Refactor > 1 day or > 2 primitives | Aligns with 18.2 being a single-wave gap closure. | ✓ |
| Refactor crosses Phase 18 region API | Fall back only if the real fix would change the `region: String` prop convention. Tighter scope. | |
| Any refactor > 4h | Hair-trigger fallback. Risk: more cfg(ssr) debt. | |

**User's choice:** Refactor > 1 day or > 2 primitives (Recommended)
**Notes:** Cost gate enforced in plan-phase based on diagnosis findings.

### Q11: If fallback is used, how should the deferred-real-fix be tracked?

| Option | Description | Selected |
|--------|-------------|----------|
| Roadmap backlog + code comment + diagnosis section | v1.4 backlog entry + `// HYDRATE-FALLBACK: see .planning/phases/18.2/18.2-DIAGNOSIS.md` comment at the cfg(ssr) site + "Deferred real fix" section in DIAGNOSIS.md. | ✓ |
| Roadmap backlog only | Add to backlog; no inline comment. | |
| Inline TODO with assignee | Inline `// TODO(@jasper): real hydration fix`. Visible where it matters; not surfaced in planning artifacts. | |

**User's choice:** Roadmap backlog + code comment + diagnosis section (Recommended)
**Notes:** Three-place audit trail; nothing falls through.

### Q12: Should the fallback also keep the WASM render visually intact, or can it degrade Pandemonium to look like Demacia for that primitive?

| Option | Description | Selected |
|--------|-------------|----------|
| Visual parity required — SSR placeholder identical, WASM still renders the Pandemonium primitive | cfg(ssr) emits structural skeleton; WASM fills in Pandemonium content on hydrate. User sees Pandemonium as intended. | ✓ |
| Degrade to Demacia for that primitive | Both SSR and WASM render Demacia branch under Pandemonium. Visibly incomplete. | |
| Decide per primitive when fallback fires | Most flexible; least pre-committed. | |

**User's choice:** Visual parity required (Recommended)
**Notes:** Degrading to Demacia breaks Phase 18's region-variants premise.
