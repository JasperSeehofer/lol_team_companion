**Researched:** 2026-05-26

**Confidence:** HIGH on root-cause classification and fix patterns (verified against `tachys-0.2.14/src/html/mod.rs:205-220` on disk + Leptos 0.7/0.8 issue tracker). MEDIUM on per-primitive ranking (suspects flagged but multi-culprit not ruled out until bisect runs).

## Summary

The phase has a fully-locked CONTEXT (D-01..D-16) — the bisect IS the plan. Research scope is to give the planner enough material to (a) chunk D-01's short-circuit-and-reload bisect into atomic plans, (b) cite specific high-risk patterns per primitive so leaves-first order is informed not random, (c) supply concrete Leptos 0.8 fix patterns the executor reaches for AFTER the culprit is identified, and (d) specify the regression-gate Playwright spec mechanics so test authoring is mechanical.

**Root cause class (HIGH confidence):** The panic `tachys-0.2.14/src/html/mod.rs:217:14 called Option::unwrap() on a None value` lives inside `InertElement::hydrate`, where `Element::cast_from(cursor.current()).unwrap()` fires when `cursor.current()` is NOT a DOM Element node (i.e., a text node, comment, or absent sibling sits where the WASM render expected an Element). This means SSR emitted a different *count* or *kind* of DOM node than the WASM client renders for the same logical position. The trigger pattern is universal across all Pandemonium pages, so the culprit is shared — either a leaf primitive used on every page (RiotTape, Glitch, Card, SectionHead) or a shared pattern that every Pandemonium page invokes the same way.

**Primary recommendation:** Run D-01 leaves-first. Top-3 suspects ranked below; if all three short-circuit clean and the panic persists, escalate per D-04 to WASM-side `console::log` render-tracing. Real Leptos fix preferred per D-09; the canonical patterns are listed in §"Fix Patterns" with line-cited examples.

### Tachys panic site (verified from source on disk)

```rust
// /home/jasper/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/tachys-0.2.14/src/html/mod.rs
// Lines 205-220 — impl RenderHtml for InertElement, fn hydrate
fn hydrate<const FROM_SERVER: bool>(
    self,
    cursor: &Cursor,
    position: &PositionState,
) -> Self::State {
    let curr_position = position.get();
    if curr_position == Position::FirstChild {
        cursor.child();
    } else if curr_position != Position::Current {
        cursor.sibling();
    }
    let el = crate::renderer::types::Element::cast_from(cursor.current())
        .unwrap();                                                         // ← LINE 217
    position.set(Position::NextChild);
    InertElementState(self.html, el)
}
```

**What this tells us:** the failure isn't generic — it's specific to `InertElement`. `InertElement` is the tachys representation of an `inner_html=...` element OR a server-side-only HTML string blob. Anywhere the page emits inner_html OR anywhere the position-state machine walks past a "should be an element" cursor position into a text/comment/null node, this exact panic fires.

**Our `inner_html` usage:** the only direct `inner_html=` in our codebase is `src/components/region/chrome.rs:193` (`Icon` component) — but Icon is region-neutral (same structure for both regions), so it can't be the direct trigger of a Pandemonium-only mismatch unless it's invoked in a region-branching context that places it at a different sibling index.

**More likely cause:** the cursor walks off-position because a *prior* sibling produced a different node count between SSR and WASM. Anywhere a component returns `view! {}` (empty fragment) vs `view! { ... }.into_any()` (real markup) — or anywhere `Option<AnyView>` is `Some` SSR but `None` hydrate (or vice versa) — the downstream cursor desync lands on an `InertElement` later in the tree.

### Anti-Patterns to Avoid (in 18.2 fixes — derived from Leptos 0.8 hydration rules)

- **Returning `view! {}` from one branch of `if/else.into_any()`** — empty fragments produce zero DOM nodes SSR but may produce a comment marker WASM-side, causing the cursor walk to desync. Use `view! { <></> }` (Leptos 0.8 explicit empty fragment) or a stable `<span></span>` placeholder instead.
- **`is_pandemonium.then(|| view!{ ... }.into_any())`** — returns `Option<AnyView>`. Both regions see the same `is_pandemonium` value at SSR (true under Pandemonium cookie) and at hydrate (also true), so in theory this is safe. But the SSR-side `Option<AnyView>` rendering and hydrate-side rendering have historically differed in node-count handling when the option is `None`, per leptos-rs#3583. The safer canonical pattern is `if is_pandemonium { view!{ ... }.into_any() } else { view!{ <></> }.into_any() }` with identical structure.
- **`move || if is_pandemonium { branch_a.into_any() } else { branch_b.into_any() }`** — when both arms have different child counts and the outer caller doesn't wrap in an element, the cursor walks the parent's children and hits a count mismatch at the LAST child position. Both arms should produce structurally similar shapes OR the outer caller should wrap in an element that absorbs the count difference.
- **Conditional inside `Card`/`Glitch` `ChildrenFn`** — the children closure runs in both SSR and hydrate; if it produces different node counts (e.g., embedded `is_pandemonium.then(...)` inside a `Card` child slot), the cursor desync is doubly nested and hard to spot.
- **Multi-line `{move || items.iter().map(...).collect_view()}` next to literal text or other `{move || ...}` closures** — per leptos-rs#3583, Vec<AnyView> adjacent to text is a known panic source.
- **`view! { <span></span> }` as a "nothing" placeholder when surrounding context is a fragment** — generates a sibling Element on hydrate but may collapse SSR. Prefer the same `view! { <></> }` empty fragment on both arms OR a stable element on both arms.

## Common Pitfalls

### Pitfall 1: Suspense-timing race contaminating the bisect signal

**What goes wrong:** The 18.1-03 sweep noted that `/profile` Pandemonium occasionally serves a partial SSR render (`<!--s-4-o-->` Suspense placeholder + skeleton loaders) on cold first fetch while the same URL on Demacia returns fully-resolved HTML. If the bisect short-circuits a primitive and the first page reload happens to hit a Suspense-cold cache, the resulting panic could be misattributed to the just-short-circuited primitive when in fact a Suspense fallback is the real divergence point.

**Why it happens:** SSR Suspense in Leptos 0.8 streams resolved-fragment HTML after the shell ships. A cold-fetch race window can produce different markup than a warm fetch. The 18.1-03 evidence file explicitly calls this out as a pre-existing footgun.

**How to avoid:** Per CONTEXT specifics, warm-fetch each Pandemonium page once before observing the panic state. Two fetches with `wait_for_server.sh` between primitive cycles. Document the warm-up step in `18.2-DIAGNOSIS.md` so future verifiers know.

**Warning signs:** Same primitive short-circuit gives panic on one reload and no panic on the next reload without any source change.

### Pitfall 2: Mistaking a side-effect of the fix for the fix itself

**What goes wrong:** If a primitive short-circuit "fixes" the panic but the primitive itself isn't the root cause — it just happens to mask the cursor desync by removing one source of structural variance — the executor may declare victory prematurely. Restoring the primitive after declaring "fixed" reproduces the panic.

**Why it happens:** Bisect-by-removal can collapse multiple confounding causes into one apparent cause if the structural primitives interact.

**How to avoid:** Before declaring a culprit "found," re-enable the primitive with the proposed fix applied and confirm the panic STILL doesn't fire. CONTEXT D-09 (real Leptos fix) implicitly requires this; D-10 (fallback) does too — the fallback must let the primitive be re-enabled. Add this as an explicit "regression check" gate in the diagnosis flow.

**Warning signs:** "Removing X fixed it" without "reshaping X and re-adding X keeps it fixed."

### Pitfall 3: WASM event handlers attach but ALL fire on the WRONG element

**What goes wrong:** After a near-fix, hydration completes (no panic) but the cursor desync earlier in the render placed event listeners on shifted DOM nodes. Clicks fire — but on the wrong button. The regression gate's mode-toggle assertion sees a DOM change but to a wrong/unexpected value.

**Why it happens:** tachys silently recovers from minor mismatches when the cursor can still walk SOMETHING. Listeners attach to whatever sibling is at the expected index — which may be the wrong button after a structural shift earlier in the tree.

**How to avoid:** D-14 assertion shape (poll for the SPECIFIC expected DOM marker, with a useful failure message — e.g. "mode-toggle silently inert under Pandemonium on /draft" — not just "DOM changed"). Test the *exact* expected new-state class/text, not just inequality.

**Warning signs:** Mode-toggle click changes the DOM but to a weird state (wrong button shows aria-pressed=true, or the active visual style appears on a sibling).

### Pitfall 4: Pandemonium hydration succeeds on the bisect page but fails on a different Pandemonium page

**What goes wrong:** D-02's leaves-first heuristic short-circuits a primitive and observes panic clears on `/solo`. Executor commits the fix. Later regression sweep finds `/team/dashboard` Pandemonium still panics — a per-page Pandemonium view (PandemoniumTeamDashboard 7-section rebuild, or TeamGameDayBriefView) was a SECOND independent culprit.

**Why it happens:** CONTEXT D-04 + Claude's Discretion note explicitly call out "Multiple culprits handling" — if bisection finds >1 culprit primitive, plan-phase decides whether to fix all in one plan or sequence.

**How to avoid:** Per CONTEXT D-07, the regression spec must sweep ALL Pandemonium scoped routes (not just the one used in the bisect). The new `hydration-no-panic.spec.ts` runs every page × pandemonium and any one failing route reopens the bisect.

**Warning signs:** D-07 sweep passes 6 of 7 Pandemonium routes. The 1 failing route is an independent culprit and needs its own bisect cycle.

### Pitfall 5: Demacia regression introduced by the fix

**What goes wrong:** The fix reshapes a primitive (e.g., changes `Card`'s `variant="zine"` branch to share node count with `variant="gilt"`). Pandemonium now hydrates clean. But the reshape inadvertently shifts a Demacia node and the SAME tachys panic now fires on Demacia pages.

**Why it happens:** Region primitives have entangled SSR/hydrate trees per region; an "alignment" fix can shift one region's tree out of agreement.

**How to avoid:** CONTEXT D-07 mandates the regression spec runs BOTH regions. D-16 makes Demacia parity explicit. Per CONTEXT D-01..D-03, the diagnosis file records that the post-fix sweep cleared both regions.

**Warning signs:** Pandemonium suddenly works; Demacia (which worked pre-fix) now panics. Run the regression spec early in the fix cycle, not just at the end.

### Pitfall 6: Bisect commit churn leaks short-circuited code into main

**What goes wrong:** Short-circuit-and-reload cycles edit primitive source files. If a wave-1 commit goes to main with a primitive still short-circuited (returning Demacia branch from the Pandemonium arm), users see Demacia on a Pandemonium page. Worse, the panic IS fixed, so the regression gate misses it.

**Why it happens:** Bisect commits feel cheap; one can leak into a finalize-merge without re-verification.

**How to avoid:** Final commit before plan close-out MUST restore every short-circuited primitive AND apply the real fix to the identified culprit. The regression spec then validates the restored-and-fixed state. Add an explicit "no `return view! { Demacia branch }.into_any();` in is_pandemonium arm" grep to the verification.

**Warning signs:** Pandemonium pages now render identically to Demacia (Demacia branch leaking through) instead of showing RiotTape + Glitch + zine layout.

## Code Examples

### Fix Pattern 1: align node counts between region branches

**Problem:** `Card` variant="zine" emits 4 absolutely-positioned bracket-corner divs + children. `variant="default"` (still Pandemonium) emits children only. The branches differ by 4 nodes when both render under Pandemonium. If a parent renders `if condition { Card variant=zine } else { Card variant=default }`, the cursor walk fails when condition flips.

**Fix:** Have both branches produce identical node count via an always-rendered conditional inside a single Card shape:

```rust
// CANONICAL FIX PATTERN (sourced from leptos-patterns rule 19 + Leptos 0.8 hydration docs)
view! {
    <div class="bg-surface border border-outline/30 rounded-none p-6 relative">
        // Always emit the 4 corner anchor positions; hide via CSS when not zine
        <div class=move || if var_is_zine {
            "absolute top-0 left-0 w-3 h-3 border-l-2 border-t-2 border-accent"
        } else { "hidden" }></div>
        // ... repeat for tr/bl/br ...
        {children()}
    </div>
}.into_any()
```

`[CITED: book.leptos.dev/ssr/24_hydration_bugs.html]` — node count must match between SSR and hydrate; CSS hide is safe because the DOM still has the node.

### Fix Pattern 2: replace `view! {}` empty fragment with stable placeholder

**Problem:** `view! {}` (used at `src/components/region/layout.rs:43`, `:67`, and `src/components/draft_board.rs:146`, `:159`, `:177`, `src/pages/draft.rs:4018`) is the "empty branch" placeholder. Per leptos-rs#3583, empty fragments adjacent to other fragments can desync the cursor walk.

**Fix:** Replace `view! {}` with `view! { <></> }` (Leptos 0.8 explicit Fragment) and verify the surrounding context is an element wrapper. If the parent's children are `[text]{view!{}}[text]` shape — wrap children in a `<span>` so adjacency isn't text-fragment-text:

```rust
// BEFORE (suspect):
{if is_pandemonium {
    view! { <span class="font-mono text-[10px] text-muted leading-none">"// BAN"</span> }.into_any()
} else {
    view! {}.into_any()                                               // ← suspect: empty fragment
}}

// AFTER (safer):
{if is_pandemonium {
    view! { <span class="font-mono text-[10px] text-muted leading-none">"// BAN"</span> }.into_any()
} else {
    view! { <span></span> }.into_any()                                 // ← stable element placeholder
}}
```

`[CITED: leptos-rs/leptos#3583]` — Vec<AnyView> with empty/text-only output between text nodes triggers the same panic class.

### Fix Pattern 3: prefer `<Show fallback=`...> over `if/else.into_any()` when reactivity is involved

**Problem:** `move || if signal.get() { A.into_any() } else { B.into_any() }` re-runs on signal change and swaps fragments. Hydration sees the SSR-initial branch, then the WASM-restored signal may flip the branch. If A and B differ in node count, the swap can desync the cursor on the next sibling walk.

**Fix:** `<Show when=move || signal.get() fallback=move || B>{move || A}</Show>` — Show is designed to handle the swap with hydrate-safe semantics.

```rust
// BEFORE (suspect):
{move || if is_pandemonium {
    view! { <RiotTape label="DRAFT_LEDGER · v0.1" /> }.into_any()
} else {
    view! { <HeraldicDivider /> }.into_any()
}}

// AFTER (Show-based):
<Show
    when=move || is_pandemonium
    fallback=|| view! { <HeraldicDivider /> }
>
    <RiotTape label="DRAFT_LEDGER · v0.1" />
</Show>
```

**Caveat:** In our codebase, `region` is a String captured ONCE at page entry (per CONTEXT canonical_refs "Region prop sourcing"). It does NOT change reactively per-page-instance. So `Show` and `if/else.into_any()` should behave identically in theory. If they don't, that's evidence of a deeper Show vs branch-swap implementation difference — flag for D-04 escalation.

### Fix Pattern 4: SSR-only placeholder under D-10 fallback

**Problem:** D-10 fires if real fix > 1 day or requires reshaping > 1 primitive's signature. Need an SSR-shape that the WASM client agrees with.

**Fix:** Gate the structural-rich SSR markup behind `cfg(feature = "ssr")` to emit a same-skeleton placeholder, while the WASM client (`hydrate` feature) renders the full Pandemonium content post-hydrate:

```rust
// EXAMPLE (worked, NOT for execution unless D-10 fires):
#[component]
pub fn RiotTape(
    #[prop(optional, default = 200)] width: u32,
    #[prop(optional, default = "RIOT")] label: &'static str,
) -> impl IntoView {
    #[cfg(feature = "ssr")]
    {
        // SSR placeholder — same outer div, empty children. WASM will replace.
        let style = format!("width: {}px; height: 22px; position: relative;", width);
        return view! { <div style=style></div> }.into_any();
    }

    #[cfg(feature = "hydrate")]
    {
        // Original rich rendering — full RiotTape with label, color, rotation.
        let style = format!(
            "width: {}px; height: 22px; position: relative; \
             background: var(--accent-3); transform: rotate(-1.2deg); \
             box-shadow: 0 1px 0 rgba(0,0,0,0.4), 2px 2px 0 var(--color-accent), -2px -1px 0 var(--accent-2); \
             overflow: hidden;",
            width
        );
        return view! {
            <div style=style>
                <div
                    class="absolute inset-0 flex items-center justify-center font-mono text-[11px] font-bold uppercase tracking-[0.18em]"
                    style="color: var(--t-accent-contrast); mix-blend-mode: multiply"
                >
                    {format!("//// {} //// {} //// {} ////", label, label, label)}
                </div>
            </div>
        }.into_any();
    }

    // Both cfg arms must produce same outer-element shape.
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Leptos 0.6 ID-based hydration (each element had a unique server-assigned ID) | Leptos 0.7+ tachys-based cursor walk | 0.6 → 0.7 (Sep 2024) | Cursor walk is faster but unforgiving: any node-count mismatch desyncs. The 18.1-HYDRATE-01 panic is a direct artifact of this newer model. `[CITED: book.leptos.dev/ssr/24_hydration_bugs.html]` |
| `view! {}` as empty placeholder | `view! { <></> }` explicit fragment OR stable element | 0.7+ | Empty fragments are still legal but discouraged adjacent to other fragments. |
| `if/else.into_any()` | Still legal; `<Show fallback=...>` preferred for reactive swap | 0.7+ | `.into_any()` documented in leptos-patterns rule 19; `Show` introduced as the canonical reactive-swap primitive. |

**Deprecated/outdated:**
- Leptos 0.6 unique-ID hydration: gone. Don't try to retro-fit `data-hk` attributes — they're ignored in 0.7+.
- `tachys 0.2.9` (referenced in older issue tracker as line 188) — superseded by 0.2.14; line numbers shifted to 217. Verified by reading the actual `~/.cargo/registry/.../tachys-0.2.14/src/html/mod.rs:217` on disk.
