# Phase 18.2: Code Review Report

**Reviewed:** 2026-05-26
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

The core fix is sound: the hydrate-side `provide_context(InitialTheme(...))` in `App()` body runs before the `view!` macro instantiates `<Routes>`, the `<html data-theme>` attribute is validated against the same `{"demacia", "pandemonium"}` allowlist as the SSR `theme_layer`, and the fallback string matches the SSR fallback in `shell()`. The `CompanionSigil` refactor to take `region: String` as a prop is correctly mirrored at every call site under review (nav.rs:312, login.rs:91, register.rs:122, closed_beta.rs:69, draft.rs:1583). Owner-scope timing is fine — context provided in a component body propagates to children rendered in the same `view!`.

There are no security or correctness blockers. However, the new regression spec has meaningful coverage gaps and one weak assertion that could let a future regression slip through:

1. The panic-sweep matrix omits the three routes whose call sites the refactor actually touched (`/auth/login`, `/auth/register`, `/closed-beta`).
2. `/match/test-match-id` likely 404s or errors out before WASM hydrates the panic surface, hiding a regression behind an error template.
3. The draft slot click-fill test's final assertion is `text=Ahri` with `.first()`, which can pass even when the slot is never filled (the search input itself contains "Ahri").
4. The hydrate-side provider silently swallows malformed/missing `data-theme` and defaults to "demacia" — fine in practice but worth a `tracing::debug!` so the silent-divergence failure mode is observable in dev.

Detailed findings below.

## Warnings

### WR-01: Panic-sweep matrix misses every route that the refactor actually touched

**File:** `e2e/tests/hydration-no-panic.spec.ts:72-80`

**Issue:** The fix in Plan 03 explicitly updated `CompanionSigil` call sites in `src/pages/auth/login.rs`, `src/pages/auth/register.rs`, `src/pages/closed_beta.rs`, `src/components/nav.rs`, and `src/pages/draft.rs`. The new regression spec's `SCOPED_ROUTES` covers `/solo`, `/draft`, `/team/dashboard`, `/tree-drafter`, `/champion-pool`, `/post-game`, and `/match/test-match-id`, but does NOT cover `/auth/login`, `/auth/register`, or `/closed-beta`. These three routes are exactly the surfaces where a regression of the CompanionSigil refactor would surface (e.g. someone re-introduces `use_context::<InitialTheme>()` inside CompanionSigil, or someone drops the `region` prop wiring on these specific pages). The auth/closed-beta pages also render before the user has a chance to hit any other route in the test matrix, so they have the highest blast radius for a hydration panic regression. The phase-context claim that this spec "actually catches a regression of the original bug" is only partially true — it catches regressions on the AUTHED routes (which the original bug also hit), but not on the public-facing surfaces that the refactor touched.

Note: `authedPage` already navigates through `/auth/register` to establish the session, so attaching panic listeners before that navigation and asserting `errors.toEqual([])` would catch the auth-flow case for free. The closed-beta case needs an unauthenticated `page` fixture (or `setRegion` before the page registers).

**Fix:**
```typescript
// Add public-route variant of the panic sweep, using the plain `page`
// fixture (not authedPage) so /auth/* and /closed-beta render in their
// unauthenticated states.

const PUBLIC_SCOPED_ROUTES = [
  "/auth/login",
  "/auth/register?invite=E2E-TEST",
  "/closed-beta",
] as const;

for (const region of ["demacia", "pandemonium"] as const) {
  for (const url of PUBLIC_SCOPED_ROUTES) {
    test(`${url} hydrates without tachys panic [${region}]`, async ({ page }) => {
      const errors = await attachPanicListeners(page);
      await setRegion(page, region);
      await page.goto(url);
      await page.waitForLoadState("networkidle");
      await page.waitForTimeout(1500);
      expect(
        errors,
        `Expected no panic on ${url} [${region}]; got: ${errors.join("\n")}`
      ).toEqual([]);
    });
  }
}
```

### WR-02: `/match/test-match-id` likely renders an error/empty state instead of the real region-branching surface

**File:** `e2e/tests/hydration-no-panic.spec.ts:79`

**Issue:** `MatchDetailPage` calls `get_match_detail("test-match-id")` (see `src/pages/match_detail.rs:10`). The literal string `"test-match-id"` is not a valid Riot match ID and almost certainly does not exist in the SurrealDB `match_detail` cache for a freshly-registered e2e user. The page will resolve to either (a) a server-fn error caught by `Suspense`'s error boundary, or (b) an empty state that never renders the region-branching primitives the test is supposed to exercise. Either way, this entry in `SCOPED_ROUTES` provides ~zero coverage value — a true regression of the original `tachys` panic would happen during the hydration of the region-branching primitives (`<Card>`, `<Glitch>`, `<SectionHead>`, etc.), and those don't render at all on an error/empty page.

Worse, this gives a false sense of security: the test will continue to pass even after a hypothetical regression on `/match/:id` because the panicky code path is never reached.

**Fix:** Either (a) remove `/match/test-match-id` from `SCOPED_ROUTES` and document the omission with a TODO referencing the need for a seeded match fixture, or (b) seed a deterministic test match into the DB in a global setup and navigate to its real ID:

```typescript
// Option A — remove until a fixture exists:
const SCOPED_ROUTES = [
  "/solo",
  "/draft",
  "/team/dashboard",
  "/tree-drafter",
  "/champion-pool",
  "/post-game",
  // TODO(18.3): re-add "/match/<seeded-id>" once an e2e match fixture exists.
] as const;
```

### WR-03: `draft slot click-fill` assertion can pass without the slot ever being filled

**File:** `e2e/tests/hydration-no-panic.spec.ts:298-300`

**Issue:** The test fills the champion search input with `"Ahri"` (line 286), then clicks the Ahri grid button (line 291), then asserts `expect(authedPage.locator('text=Ahri').first()).toBeVisible({ timeout: 2000 })`. The problem: by the time this assertion runs, "Ahri" is ALREADY visible on the page — the search input contains the string "Ahri" (`prop:value` reflects it back into the DOM), and the champion-grid button itself has `alt="Ahri"` plus a visible label. `locator('text=Ahri').first()` resolves to one of those existing nodes, not to a slot fill. The assertion will pass even if the click handler is broken, panics silently, or never fires — defeating the purpose of D-14's "specific observable change" requirement that the test's own header comment cites (line 27 and lines 174-176).

The comment at line 298 acknowledges "there are multiple 'Ahri' hits" but treats this as a feature; it is a bug. The "observable change" must be a NEW node that appears only after the click, not any existing node.

**Fix:** Target a node that exists only post-click — the slot's filled state. Looking at the draft board, a filled slot renders the champion image with a `data-slot-filled` style or a delete/clear button next to the slot. A more specific assertion would be the appearance of a clear/delete button inside the targeted slot, or counting slot images:

```typescript
// Capture pre-click filled-slot count, then assert it increased by 1.
const filledBefore = await authedPage
  .locator('[data-testid="draft-slot"][data-filled="true"]')  // or equivalent
  .count();
await ahriBtn.first().click();
await expect(async () => {
  const filledAfter = await authedPage
    .locator('[data-testid="draft-slot"][data-filled="true"]')
    .count();
  expect(filledAfter).toBe(filledBefore + 1);
}).toPass({ timeout: 2000 });
```

If no stable test-id exists, add one in `draft_board.rs` — a single `data-filled={champion.is_some()}` attribute on each slot wrapper is sufficient. As-is, the test provides false confidence.

### WR-04: Silent fallback when `<html data-theme>` is missing or out of allowlist hides the failure mode the fix is designed to prevent

**File:** `src/app.rs:116-125`

**Issue:** The hydrate-side provider chain is:

```rust
let theme = web_sys::window()
    .and_then(|w| w.document())
    .and_then(|d| d.document_element())
    .and_then(|el| el.get_attribute("data-theme"))
    .filter(|v| v == "demacia" || v == "pandemonium")
    .unwrap_or_else(|| "demacia".to_string());
provide_context(InitialTheme(theme));
```

Every failure mode collapses silently to `"demacia"`:
- `web_sys::window()` returns None (impossible in browser, but cfg(hydrate) doesn't enforce browser)
- `document_element()` returns None
- `get_attribute("data-theme")` returns None (SSR forgot to render it)
- The attribute value is not in the allowlist (DevTools tamper, browser extension injection, race with ThemeToggle)

If SSR renders `data-theme="pandemonium"` but for any reason the hydrate-side read fails — say, an out-of-order script that strips the attribute, or a Tailwind/CSS-in-JS plugin that mutates it — we land back in the EXACT failure mode the fix is designed to prevent: SSR rendered Pandemonium structure, WASM hydrates with `"demacia"` context, primitives diverge, `tachys` panics. The user observes a regression of HYDRATE-01 with no diagnostic signal.

Note this is not a bug in the fix itself — the fallback is correct. The problem is observability. Without instrumentation, a future regression will look identical to the bug we just fixed and waste another investigation cycle.

**Fix:** Log a warning when the read fails. This is a one-line change that gives any future regression a clear breadcrumb:

```rust
#[cfg(feature = "hydrate")]
{
    let raw = web_sys::window()
        .and_then(|w| w.document())
        .and_then(|d| d.document_element())
        .and_then(|el| el.get_attribute("data-theme"));

    let theme = match raw.as_deref() {
        Some(v @ ("demacia" | "pandemonium")) => v.to_string(),
        other => {
            web_sys::console::warn_1(
                &format!(
                    "[hydrate] InitialTheme read failed (got {other:?}); \
                     falling back to \"demacia\". This will cause an \
                     SSR/WASM divergence if SSR rendered Pandemonium."
                ).into()
            );
            "demacia".to_string()
        }
    };
    provide_context(InitialTheme(theme));
}
```

The `console.warn` is filtered out by `PANIC_REGEX` in the test spec (D-08), so it does not false-positive the regression gate, but it does surface in dev console + `pageerror` instrumentation.

## Info

### IN-01: `CompanionSigil` prop type `String` forces a clone at every call site

**File:** `src/components/region/ornaments.rs:151`

**Issue:** The prop is `#[prop(into)] region: String`. Every call site does `region=region.clone()` (nav.rs:312, login.rs:91, register.rs:122, closed_beta.rs:69, draft.rs:1583), which is a heap allocation per render. Since the only thing the body does with `region` is `region == "pandemonium"` (line 153), borrowing `&str` would be sufficient and avoid the clones. However, this would require either a lifetime parameter on the component (awkward in Leptos) or matching the wider project convention.

Worth noting: the rest of the region primitive family (`Card`, `Glitch`, `SectionHead`, etc., per draft.rs:4) all take `String`, so this is consistent with the project convention. Filing as info, not warning, because the inconsistency cost of changing only `CompanionSigil` would exceed the per-render cost.

**Fix:** No action recommended. If a future refactor sweeps all region primitives to `&'static str` via an enum or `Region` newtype, include `CompanionSigil`.

### IN-02: `PANIC_REGEX` matches `"tachys"` anywhere — could false-positive a benign log line

**File:** `e2e/tests/hydration-no-panic.spec.ts:64`

**Issue:** `const PANIC_REGEX = /panicked at|Option::unwrap.*None|tachys/i;` — the unanchored `tachys` clause matches any `console.error` containing the word "tachys" in any context, including a future log line like `"[debug] tachys hydration started"`. The phase context describes this as "any other tachys-namespaced runtime error," which is intentional. But it is a brittle choice: if `leptos`/`tachys` ever changes the log namespace conventions, this will start false-positiving and authors will lower the regex strictness to make tests green, which loses signal.

**Fix:** Tighten to known-bad patterns from the evidence trail. The original panic was `Option::unwrap()` in `tachys::html`, so:

```typescript
const PANIC_REGEX = /panicked at|Option::unwrap.*None|tachys[^\s]*panic|tachys[^\s]*hydrate.*(panic|unwrap)/i;
```

Not critical — the current regex is fine for v1 of this gate.

### IN-03: Effect cycle risk on `auth_user` refetch in `DraftPage`

**File:** `src/pages/draft.rs:954-959, 965-969`

**Issue:** Two effects interact:

1. Lines 954-959: when `auth_user` loads, resolve mode and set `mode_current`.
2. Lines 965-969: when `set_mode_action` succeeds, `auth_user.refetch()`.

If the user clicks the mode toggle, `set_mode_current.set(new_mode)` fires (line 971), `set_mode_action` dispatches, and on success `auth_user.refetch()` produces a new resource value. The first effect then runs again and re-sets `mode_current` to the same value. Leptos 0.8 signal equality should prevent the second effect from re-triggering (set with same value), so this is not a runtime bug — but it is a fragile arrangement that depends on signal de-duplication semantics.

This is out of scope for the 18.2 hydration fix; flagging because the file is in this review's scope.

**Fix:** No action recommended for this phase. If the cycle ever produces flicker or double-saves, suppress the resync effect during in-flight mode actions:

```rust
let mode_action_pending = move || set_mode_action.pending().get();
Effect::new(move |_| {
    if mode_action_pending() { return; }
    if let Some(Ok(Some(user))) = auth_user.get() {
        let resolved = resolve_mode(&user.draft_mode, &region_for_mode, "draft");
        set_mode_current.set(resolved);
    }
});
```

### IN-04: Hydrate-side `data-theme` reader could be a single helper in `theme_layer`-adjacent module

**File:** `src/app.rs:116-125`

**Issue:** The hydrate-side read of `<html data-theme>` is inlined in `App()` body. If another component ever needs to perform the same read (e.g. a worker, devtools panel, future ThemeToggle reset), the chain would be duplicated. The SSR side already has a shared `theme_layer::parse_theme_cookie` + `resolve_theme`. A symmetric hydrate-side helper would keep both sides of the fence using the same allowlist constant.

**Fix:** Extract to a `theme_hydrate.rs` (or co-locate in `theme_layer.rs` behind `#[cfg(feature = "hydrate")]`) so the allowlist `["demacia", "pandemonium"]` lives in one place:

```rust
// src/server/theme_layer.rs (or new src/theme_hydrate.rs guarded by cfg(hydrate))
#[cfg(feature = "hydrate")]
pub fn read_hydrate_theme() -> InitialTheme {
    let raw = web_sys::window()
        .and_then(|w| w.document())
        .and_then(|d| d.document_element())
        .and_then(|el| el.get_attribute("data-theme"));
    let theme = match raw.as_deref() {
        Some(v) if VALID_THEMES.contains(&v) => v.to_string(),
        _ => "demacia".to_string(),
    };
    InitialTheme(theme)
}
```

Then `app.rs` becomes `provide_context(read_hydrate_theme());`. Reuses `VALID_THEMES` from the SSR side so the allowlist cannot drift between the two paths.
