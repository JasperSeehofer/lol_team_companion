    - "SSR-only context bridging via DOM attribute serialization: the canonical pattern for any per-request value that the SSR-only leptos_routes_with_context closure provides (theme, locale, feature flags, user preferences) is to render it onto an HTML attribute the SSR shell controls (e.g., `<html data-theme=...>`, `<body data-locale=...>`), then on WASM hydrate re-read the attribute via `web_sys::window().and_then(|w| w.document()).and_then(|d| d.document_element()).and_then(|el| el.get_attribute(\"data-X\"))` and re-provide via `provide_context` BEFORE the `view!` macro runs. The attribute is the serialization channel; `provide_context` at the top of `App::App()` is the bridge."
    - "Owner-scope reasoning: `provide_context` writes to the CURRENT Leptos owner; `use_context` walks the parent chain UP. Placement at the top of `App::App()` body (before the `view!` macro) is sufficient for every descendant including `<Router>` / `<Routes>` / Route view children — they create child owners that walk back up to the App owner. The Plan 03 hypothesis that Routes children needed their own provide site was incorrect; the prototype likely failed due to a stale WASM bundle (Plan 03 documented a stale leptos daemon problem in its own Deviation 3)."

## Decisions Made

See key-decisions in frontmatter. Four pivotal decisions:

1. **Pick App-body provide site over the alternatives.** Read `<html data-theme>` via `web_sys` is the smallest diff, mirrors the SSR-side serialization channel exactly, and avoids re-implementing the cookie precedence rules on the WASM side.
2. **Eager top-of-body read, not Effect-deferred.** An Effect would run AFTER the first hydrate walk and would not be visible during the panic's microtask. Plan 03's prototype may have hit exactly this trap.
3. **Defensive allowlist filter even though `theme_injection_middleware` already validates.** Belt-and-braces: any future DOM-injection path that mutates `<html data-theme>` cannot smuggle unexpected strings into InitialTheme.
4. **Spec lives in its own file, listener stays spec-local.** Per D-06 — don't slow every other e2e test with a panic listener it doesn't need; per D-08 — narrow PANIC_REGEX, not the generic captureErrors from helpers.ts.
