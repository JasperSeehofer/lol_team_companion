<objective>
Close the architectural root cause of UI-18-RUNTIME-01: `src/main.rs:88` hardcodes `provide_context(InitialTheme::default())`, so every Phase 18 region-branching page reads `region = "demacia"` regardless of what the user has selected. This plan replaces that hardcode with a per-request theme injection layer and adds a synchronous cookie write to the toggle so the choice survives reload.

**Surface:**
1. New `src/server/theme_layer.rs` axum `tower::Layer` (~80–120 LOC) that:
   - Parses the `Cookie` header for `lol_companion_theme=<value>`
   - Validates the value against the `{"demacia", "pandemonium"}` allowlist (D-05)
   - Reads `AuthSession` from request extensions (populated by `auth_layer` running before us, D-03)
   - Resolves theme precedence: valid cookie > `AuthSession.user.theme` > `"demacia"` (D-02)
   - Inserts the resolved `InitialTheme(String)` into request extensions for the leptos context closure
2. `src/server/mod.rs` adds `pub mod theme_layer;`
3. `src/main.rs` chains the new layer AFTER `.layer(auth_layer)` and updates the leptos context closure to read `InitialTheme` from request extensions via `leptos_axum::extract_with_state` or by reading the extension through `leptos_axum`'s sync request-context API. Per the existing comment at main.rs:82–87, the sync closure cannot await an extractor — but it CAN call `use_context::<leptos_axum::ResponseOptions>` style synchronous accessors. The simplest path: have the layer insert `InitialTheme` directly, and have the closure use `leptos_axum::extract::<axum::Extension<InitialTheme>, ()>` is async — so instead use the existing `provide_context(...)` flow with a request-cloned extension read via `leptos_axum`'s `LeptosOptions`-style runner. **Concrete approach (D-03):** the layer inserts `axum::Extension(InitialTheme(...))` into request extensions; the leptos `with_context` closure receives the request via its closure capture pattern — see the existing `app_state.db` provide_context for the precedent. Plan task 1 chooses between (a) wrapping `leptos_routes_with_context` so each request gets a fresh closure with the right `InitialTheme`, or (b) reading the extension synchronously inside the existing closure by exposing the request in scope. Per leptos_axum 0.8, the recommended pattern is `leptos_routes_with_context` where the context closure receives no request directly — therefore the layer must put `InitialTheme` somewhere the closure can synchronously read it. The Leptos 0.8 idiom: the closure runs INSIDE the request scope and can read request extensions via `leptos::Owner::current()` extensions, OR we provide the value through a per-route Axum extension that `leptos_axum::extract` can pull synchronously. **Locked sub-decision in this plan:** use `tower::Layer` to insert `Extension(InitialTheme(...))` into the request, and in the leptos context closure call `expect_context::<leptos_axum::ResponseOptions>` style accessors. If the leptos 0.8 sync API does not expose request extensions, fall back to the `with_context` per-request pattern by reading the extension in a synchronous wrapper Service that calls `leptos_axum::handle_server_fns_with_context` style entry — but the SIMPLEST and most-reviewed path is: use the same `provide_context` closure, and have the closure call `leptos_axum::extract_with_state::<axum::Extension<InitialTheme>, _, _>` — which IS sync-callable in Leptos 0.8 when wrapped in `leptos::task::spawn_local` — **but we cannot spawn_local in this closure either**. **Final locked approach:** wrap `leptos_routes_with_context` in a custom `service_fn` (or `from_fn`) middleware applied OUTSIDE the leptos routes that, on each request, reads `Extension(InitialTheme)` from the request and stashes it into a per-task `tokio::task_local!` (or thread-local) that the closure reads synchronously. Task 1 implementation will use the cleanest pattern that compiles — `tokio::task_local!` is the documented Leptos 0.8 pattern for sync request-scoped values; the layer sets it before handing off to leptos_routes.
4. `src/components/theme_toggle.rs` writes the `lol_companion_theme` cookie synchronously via `web_sys::HtmlDocument::set_cookie` in the same JS turn as the `data-theme` attribute update, BEFORE the `spawn_local` for the DB persist. This closes the race documented in CONTEXT.md risk surface.

Output: SSR provides the user's actual theme on every request; clicking the toggle persists to BOTH the cookie (instant, unauthenticated-friendly) and the DB (authoritative for authenticated users on other devices); compile both targets clean.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → axum HTTP request | Untrusted `Cookie` header crosses here; client can forge any value |
| `axum_login` auth_layer → request extensions | `AuthSession.user` is server-derived from session cookie (trusted) |
| theme_layer → leptos context | Theme value flows into SSR-rendered `<html data-theme="...">` attribute and into `region: String` props on every scoped page |
| `web_sys::HtmlDocument::set_cookie` (browser) | Client-set cookie is round-tripped to the next request; trust depends on validation at the receiving boundary (theme_layer) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18.1-01-01 | T (Tampering) | `lol_companion_theme` cookie value | mitigate | `parse_theme_cookie` validates against `["demacia", "pandemonium"]` allowlist BEFORE returning `Some(...)` per D-05. Invalid/forged values (XSS strings, SQL fragments, arbitrary text) are treated as cookie absent → resolver falls through to auth or default. Validated value is interpolated only into Tailwind class selectors and `data-theme` attribute — never into raw HTML, never into a DB query, never into a redirect URL. The two allowlisted values are both URL-safe and HTML-attribute-safe. |
| T-18.1-01-02 | I (Information disclosure) | Cookie not marked `HttpOnly` | accept | D-01 explicitly locks `HttpOnly=false` because the client must read the cookie for optimistic update after toggle click. The cookie value is a 2-element enum string (`demacia`/`pandemonium`); it is NOT an auth token, session ID, or PII. JS access yields no escalation; if an attacker can run JS in the page they already have full session control regardless of cookie flags. |
| T-18.1-01-03 | T (Tampering) | Cookie write race (sub-100ms navigation between click and cookie landing) | mitigate | Task 1 mandates that the cookie write executes SYNCHRONOUSLY in the same JS turn as `data-theme` attribute update, BEFORE the `spawn_local` for the DB persist. The `document.cookie =` setter is synchronous in WebKit/Blink/Gecko (W3C HTML spec). The race window is therefore reduced from "spawn_local resolves before navigation" (variable, can be 10–500ms) to "the synchronous JS turn before the click event handler returns" (sub-1ms). |
| T-18.1-01-04 | S (Spoofing) | `AuthSession.user.theme` value | accept | Theme value comes from `AppUser.theme` which is populated from `DbUser.theme` which is constrained by `schema.surql:13`'s `ASSERT $value IN ['demacia','pandemonium']`.
