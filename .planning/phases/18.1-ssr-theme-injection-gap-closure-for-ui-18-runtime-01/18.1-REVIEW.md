# Phase 18.1: Code Review Report

**Reviewed:** 2026-05-26
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Phase 18.1 closes the SSR-side gap behind UI-18-RUNTIME-01. The core mechanism
— a tokio task-local populated by an axum middleware between the auth layer
and the leptos route context closure — is sound. Cookie parsing is safe
against tampered/oversized input (the allowlist is matched on a borrowed
slice before any allocation), the task-local scope correctly bounds the
value to a single request via `tokio::task_local::scope`, the documented
precedence (cookie > auth > default) is the one actually implemented,
layer ordering in `main.rs` is correct (auth outermost, runs first), and
the WASM `set_cookie` call follows the no-unwrap discipline from
`wasm-patterns` rule 35.

The defects below are all in the documentation / test-helper layer:
stale narrative comments in `region-diff.spec.ts` that describe pre-18.1
behaviour, a docstring in `theme_layer.rs` that contradicts the actual
builder order it ships alongside, a hardcoded `domain: "127.0.0.1"` in
`fixtures.ts` that silently breaks if `BASE_URL` is changed to
`localhost`, and an `addCookies` API claim that mis-states Playwright's
actual rule. None block the SSR fix from doing its job in production.

## Warnings

### WR-01: `setRegion` hardcodes `domain: "127.0.0.1"`, silently breaks if `BASE_URL=http://localhost:3020`

**File:** `e2e/tests/fixtures.ts:140`
**Issue:** The cookie is registered with `domain: "127.0.0.1"`. The
Playwright config at `e2e/playwright.config.ts:15` reads
`baseURL: process.env.BASE_URL ?? "http://127.0.0.1:3020"` — so any
contributor who sets `BASE_URL=http://localhost:3020` (a common dev
fallback) will get a cookie pinned to `127.0.0.1` that the browser
refuses to send to a `localhost` request. The reload then re-issues
the request with no cookie, SSR renders `demacia` by default, the
`themeAttr !== region` assertion at line 157 fires, and the failure
message ("setRegion failed: expected pandemonium, got demacia") looks
like an SSR bug rather than a host-mismatch in the helper. The
existing patch (Wave 2) chose `domain + path` over `url` for a
different reason (`about:blank` handling), but the host coupling is
now load-bearing in a way the docstring does not flag.
**Fix:** Derive the cookie domain from the configured `baseURL`
rather than hardcoding the literal:
```ts
import { test as base } from "@playwright/test";
// inside setRegion:
const baseURL = page.context()._options?.baseURL
  ?? process.env.BASE_URL
  ?? "http://127.0.0.1:3020";
const domain = new URL(baseURL).hostname; // "127.0.0.1" OR "localhost"
await page.context().addCookies([{
  name: "lol_companion_theme",
  value: region,
  domain,
  path: "/",
  sameSite: "Lax",
}]);
```
Or use Playwright's `url:` form when a real navigation has already
happened, and fall back to `domain + path` only on `about:blank`.

### WR-02: `setRegion` assertion at line 156-160 is skipped on the `about:blank` path, hiding cookie-honour regressions

**File:** `e2e/tests/fixtures.ts:149-160`
**Issue:** The "assert SSR honoured the cookie" check (`themeAttr !== region`)
is gated on `currentUrl && currentUrl !== "about:blank"`. Callers that
invoke `setRegion(page, "pandemonium")` BEFORE `page.goto(...)` get
zero verification that the cookie actually reaches the server — the
helper sets the cookie, sees `about:blank`, and silently returns
"success" even if the cookie was rejected by Playwright (e.g.
expired, wrong domain, sameSite mismatch). The next `page.goto(...)`
will SSR-render `demacia`, screenshots will match across regions,
and `pixelDiffRatio < 0.014` fires — surfacing as "regions don't
differ" rather than "cookie wasn't sent". This obscures the actual
failure mode for any future regression.
**Fix:** Move the assertion outside the `about:blank` guard.
For the pre-navigation path, defer the check until after the caller
issues their first `page.goto(...)` — track a "pending verification"
flag on the page object, or expose `verifyRegion(page, region)` that
callers can run after navigation:
```ts
// After the addCookies call:
const currentUrl = page.url();
if (currentUrl && currentUrl !== "about:blank") {
  await page.reload({ waitUntil: "networkidle" });
  await page.waitForTimeout(500);
}
// Always assert when we have a real page.
if (page.url() && page.url() !== "about:blank") {
  const themeAttr = await page.getAttribute("html", "data-theme");
  if (themeAttr !== region) {
    throw new Error(`setRegion failed: expected ${region}, got ${themeAttr}`);
  }
}
```

## Info

### IN-01: Stale narrative comments in `region-diff.spec.ts` contradict the new SSR-authoritative behaviour

**File:** `e2e/tests/region-diff.spec.ts:51-59`
**Issue:** The inline block above `captureBothRegions` still says
"Navigate first, then set region in-page (avoids SSR resetting
data-theme to demacia on reload). The server always SSR-renders
data-theme='demacia' (hardcoded in main.rs InitialTheme::default());
setRegion clicks the ThemeToggle button which immediately updates
data-theme client-side." Every clause is now false post-18.1: SSR
honours the cookie, `main.rs` is no longer hardcoded to demacia, and
`setRegion` no longer clicks a button — it sets a cookie and reloads.
The file's own header at lines 13-21 already documents the new
behaviour correctly, so the in-function block is an outdated
duplicate and will mislead the next person editing this file.
**Fix:** Replace the lines 51-59 comment with a one-liner that
references the file header, e.g.
`// See header for SSR-authoritative cookie-injection rationale.`
Drop the "always SSR-renders demacia" and "clicks the toggle"
language entirely.

### IN-02: `theme_layer.rs` docstring claims middleware goes "AFTER `.layer(auth_layer)`" but the code places it BEFORE

**File:** `src/server/theme_layer.rs:88-90`
**Issue:** The doc comment for `theme_injection_middleware` reads
`"Placement: AFTER `.layer(auth_layer)` in builder order so
`auth_layer` runs first..."` The actual builder in `main.rs:113-114`
calls `.layer(theme_injection_middleware)` BEFORE `.layer(auth_layer)`.
Both are functionally correct (axum's inside-out semantics: the last
`.layer()` wraps outermost and runs first), and `main.rs` has a
detailed comment that explains the inversion — but the literal word
"AFTER" in `theme_layer.rs` directly contradicts the literal builder
sequence in `main.rs`. A future maintainer reading only the docstring
and then "fixing" `main.rs` would break the AuthSession population
order.
**Fix:** Reword the docstring to match the builder-order convention,
e.g.:
```rust
/// Placement: this middleware must be added BEFORE `.layer(auth_layer)`
/// in builder order so axum's inside-out wrapping puts `auth_layer`
/// on the OUTSIDE (runs first), making `AuthSession` available in
/// request extensions by the time this middleware runs.
```

### IN-03: `setRegion` docstring overstates Playwright's `addCookies` API rule

**File:** `e2e/tests/fixtures.ts:126-129`
**Issue:** The comment says "Playwright's `addCookies` API rejects
passing BOTH `url` and `path` — you must pass either `url` alone
(which derives domain+path) or the explicit `domain` + `path` pair."
Playwright actually accepts cookies with `url` plus an explicit
`path`; the constraint is on `url` vs `domain` (those two collide,
not `url` vs `path`). The substantive choice (`domain + path` for
`about:blank` compatibility) is correct and well-motivated, but the
API claim is wrong and will confuse anyone who reaches for the docs.
**Fix:** Tighten the wording, e.g.
`// Playwright's addCookies API takes either url (which derives
domain) or explicit domain. We use domain+path so this works before
any page.goto() (when page.url() is about:blank).`

### IN-04: `Cargo.toml` carries no new dependency — confirm-only

**File:** `Cargo.toml`
**Issue:** The phase brief flagged "Cargo.toml may have one new dep
(cookie-parsing crate) — verify pinning is consistent with the rest
of the file." No new dependency was added; cookie parsing is done by
hand in `theme_layer.rs::parse_theme_cookie` with `&str::split(';')`
and `strip_prefix`. The handwritten parser is correct for the locked
shape (`name=value` pairs, allowlist of two values, no quoting) and
avoids a transitive-dep audit, so this is acceptable — recording
here only because the review checklist required confirmation.
**Fix:** No action required. If the cookie surface ever grows beyond
this single-key allowlist (e.g. signed values, attributes, RFC 6265
quoting), revisit and adopt the `cookie` crate at version `^0.18`
to match the ecosystem axum-extra uses.
