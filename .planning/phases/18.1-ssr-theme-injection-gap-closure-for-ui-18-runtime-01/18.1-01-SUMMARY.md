# Phase 18.1 Plan 01: SSR Theme Injection Middleware Summary

Wired a per-request `InitialTheme` SSR injection middleware on top of `axum_login`'s `auth_layer` so the Phase 18 region-branching components actually receive the user's chosen region at SSR time, and made `ThemeToggle` write the `lol_companion_theme` cookie synchronously so the choice survives reload. Closes the architectural root cause of UI-18-RUNTIME-01.

## What Changed

### New: `src/server/theme_layer.rs` (198 LOC)

Three public items + one task-local:

- `pub fn parse_theme_cookie(cookie_header: &str) -> Option<String>` — splits the `Cookie:` header on `;`, scans for `lol_companion_theme=<value>`, returns `Some(value)` only if `value` is in the `{"demacia", "pandemonium"}` allowlist (D-05). All other inputs return `None` (defense against T-18.1-01-01 tampering).
- `pub fn resolve_theme(cookie: Option<String>, auth_theme: Option<String>) -> String` — precedence resolver: `cookie.or(auth_theme).unwrap_or_else(|| "demacia".to_string())` per D-02.
- `pub async fn theme_injection_middleware(req: Request, next: Next) -> Response` — axum middleware. Reads `req.headers().get(COOKIE)` and `req.extensions().get::<AuthSession>()`, calls the two resolvers, scopes `REQUEST_THEME` task-local to the resolved `InitialTheme(String)` for the duration of `next.run(req).await`.
- `tokio::task_local! { pub static REQUEST_THEME: InitialTheme; }` — the bridge that lets the synchronous `leptos_routes_with_context` closure in `main.rs` read the per-request value via `REQUEST_THEME.try_with(|t| t.clone()).unwrap_or_default()`.

### Updated: `src/main.rs`

```diff
-                provide_context(InitialTheme::default());
+                let theme = REQUEST_THEME
+                    .try_with(|t| t.clone())
+                    .unwrap_or_default();
+                provide_context(theme);
```

```diff
         .fallback(leptos_axum::file_and_error_handler::<AppState, _>(shell))
+        .layer(axum::middleware::from_fn(theme_injection_middleware))
         .layer(auth_layer)
         .with_state(app_state);
```

Builder order is `.layer(theme_layer).layer(auth_layer)` — axum tower layering applies inside-out, so the LAST `.layer()` call wraps the OUTERMOST middleware and runs FIRST. `auth_layer` runs first → populates `AuthSession` in extensions → `theme_injection_middleware` reads it → sets `REQUEST_THEME` → inner service (leptos routes) runs → closure reads `REQUEST_THEME`.

### Updated: `src/components/theme_toggle.rs`

The hydrate-only block in the `set_theme` `Callback` now:

1. **First**, writes the `lol_companion_theme` cookie synchronously via `web_sys::HtmlDocument::set_cookie` (the same JS turn as the `set_attribute("data-theme", ...)` that follows).
2. Updates `data-theme` on the document root.
3. **Then** spawns the DB persist via `leptos::task::spawn_local`.

Cookie attributes are exactly `lol_companion_theme=<value>; Path=/; Max-Age=31536000; SameSite=Lax` (D-01). No `HttpOnly` — the client must read it for the SSR-authoritative initial paint after reload.

Per wasm-patterns rule 35, every `web_sys` step is guarded with `if let Ok(...)` / `if let Some(...)` — no `.unwrap()`. Used `wasm_bindgen::JsCast::dyn_into::<web_sys::HtmlDocument>` to upcast the generic `Document`.

### Updated: `Cargo.toml`

Added `"HtmlDocument"` to the `web-sys` feature flag list (the cookie setter lives on `HtmlDocument`, not on the base `Document` trait).

### Negative-control runtime evidence (against OLD code on existing `:3020` dev server)

The pre-existing dev server at `127.0.0.1:3020` is running the OLD code (no theme middleware, unconditional `InitialTheme::default()`). All three unauthenticated cases return `demacia` regardless of cookie content — empirically reproducing the UI-18-RUNTIME-01 bug:

```
[Case A] no cookie:                         data-theme="demacia"
[Case B] cookie=pandemonium:                data-theme="demacia"  # NEW CODE will return "pandemonium"
[Case C] cookie=<script>alert(1)</script>:  data-theme="demacia"  # NEW CODE will still return "demacia" (D-05 rejection)
```

This negative control empirically confirms two things:
1. The bug exists in the OLD code exactly as documented (cookie ignored).
2. After the new middleware lands in production, Case B will flip to `pandemonium` (proven by `resolve_theme_cookie_wins_over_auth` unit test) and Case C will stay `demacia` (proven by `parse_theme_cookie_rejects_garbage` + `parse_theme_cookie_rejects_xss_string` unit tests).

## Deviations from Plan

### [Rule 4 - Environmental] Full live curl matrix (Task 2 cases d, e) deferred to verifier wave

**Found during:** Task 2 — attempted to start dev server on alternate port `:3022` from the worktree.

**Issue:** Disk filled to 100% during the `cargo leptos watch` link step. Combined target-dir footprint from the user's pre-existing main-repo dev server (~17 GB) plus the worktree's own SSR + hydrate compile artifacts (~15 GB) exhausted the 234 GB root partition before the binary could be linked. After clearing the worktree's `target/debug/incremental`, duplicate `target/debug/deps/libsurreal*.rlib` artifacts, `target/front`, and `target/wasm32-unknown-unknown` caches to recover ~3.5 GB, a second link attempt also failed (`ld.bfd: final link failed: No space left on device`).

**Why this is Rule 4 (not Rule 3):** the blocker is not a code bug — both compile-target `cargo check` invocations succeed; the issue is that the FULL link step requires more scratch space than the partition can supply with two parallel `target/` trees. Resolving this requires a structural decision (delete the main repo's `target/`, kill the user's running dev server, or move the worktree to a separate filesystem) which is outside this plan's scope.

**Mitigation applied:**
1. The 9 precedence/validation cases from the plan's Task 1 `<behavior>` block are covered by 10 unit tests (9 plan-spec + 1 bonus XSS payload) — all passing. The resolver and the cookie parser are pure functions; their correctness is fully verifiable at unit-test layer.
2. The negative-control evidence against the running OLD server proves the bug is real and that the new code's resolver path (validated above) IS the fix.
3. The layer-ordering correctness is documented in code comments at the `.layer()` call site and is the ONE remaining empirical question. The phase's Wave 3 verifier (per ROADMAP) will execute the live curl matrix as part of the broader runtime sweep, on a clean filesystem.

**Files modified to deviation:** none — this is a verification-step environmental constraint, not a code change.

### [Note] `set_user_theme` server fn return type kept as `Result<(), ServerFnError>`

The plan's Task 1 Action 4c made this a "Claude's Discretion" call. Kept as-is because:
- The cookie is already written synchronously from the client's allowlist-validated value, so the round-trip would only add latency.
- Changing the server-fn return type touches the JSON wire format and would require a coordinated client/server release (acceptable but unnecessary here).
- The validation allowlist exists at THREE layers (client gate in `set_theme` Callback, middleware in `theme_layer.rs`, SurrealDB `ASSERT $value IN [...]`) — adding a fourth via server-confirmed-return-value is defense-in-quintuple-depth, beyond diminishing returns.
