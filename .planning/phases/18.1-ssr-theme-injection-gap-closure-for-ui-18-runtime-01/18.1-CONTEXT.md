## Phase Boundary

Close the architectural gap that prevents Phase 18's structural region branching from activating at runtime. The fix is a small, focused change to the SSR context provider in `src/main.rs` plus a cookie-write side effect in `src/components/theme_toggle.rs`. No Phase-18 source code changes; no page-level changes; no new primitives.

**Touches:**
- `src/main.rs` — replace `provide_context(InitialTheme::default())` with a per-request value extracted from a `tower::Layer` middleware
- `src/main.rs` (or new module `src/server/theme_layer.rs`) — new axum middleware that reads the `lol_companion_theme` cookie and (if authenticated) `AuthSession.user.theme`, resolves precedence, and inserts the chosen theme into request extensions
- `src/components/theme_toggle.rs` — on click, write the `lol_companion_theme` cookie via `web_sys` alongside the existing DB persist (`set_user_theme` server fn)
- `e2e/tests/region-diff.spec.ts` — update `REGION_DIFF_THRESHOLD` after re-baselining (the runtime now actually flips structure; pixel diff expected to jump from 0.5–2.5% to 5–15% per scoped route)
- `e2e/tests/visual-regression.spec.ts-snapshots/authed-{draft,solo,team-dashboard,tree-drafter,champion-pool,match-detail,post-game}/**` — re-capture all scoped baselines under the new runtime; utility baselines are byte-for-byte unchanged
- `.planning/phases/18-region-variants/18-HUMAN-UAT.md` — mark UI-18-RUNTIME-01 as resolved after verification passes

**Does not include:** any change to Phase 18's region primitives, page sub-views, mode-toggle wiring, schema, DB layer, or ModeToggle component. Those all work correctly already — they just never see Pandemonium because `region` is hardcoded "demacia" upstream. Also out of scope: improving FOUC behavior on the very first paint (the cookie-injection model means SSR knows the theme before the first byte goes out for any returning visitor; first-ever visit is "demacia" by definition).

## Root cause recap

`src/main.rs:88` currently has:
```rust
provide_context(InitialTheme::default());  // always "demacia"
```

With the comment at lines 82-87 noting this is a deferred Phase-17 task. The deferral notes that closing it requires "axum middleware injection." Phase 18 built 24+ region-branching primitives + 11 scoped pages on top of `let region = use_context::<InitialTheme>().0` reads at component init. Because the context value is always "demacia", every `is_pandemonium = region == "pandemonium"` check is permanently false, and the Pandemonium structural branches never render.

Verification 2026-05-25 via agent-browser: `/draft` → click PANDEMONIUM → click LEDGER → DOM contains the Demacia `else` branch of `DraftLedgerView` ("DRAFT LEDGER" Cormorant + "BLUE SIDE — OUR FORCES" + gilt corners). The Pandemonium `if` branch markers (RiotTape "DRAFT_LEDGER · v0.1", Glitch `// BLUE_LOG` / `// RED_LOG`, `bg-scanline` zine cards) are nowhere in the DOM. Token swap works (pink/magenta accent visible); structural branching does not.

## Locked decisions

These are decisions the orchestrator committed to in the ROADMAP scope section. Plan-phase locks them as immutable; do not re-litigate during planning.

- **D-01 — Cookie name + attributes:** `lol_companion_theme=<demacia|pandemonium>; Path=/; Max-Age=31536000; SameSite=Lax`. NOT `HttpOnly` (client must read for the optimistic update after toggle click). Plan-phase confirms.
- **D-02 — Precedence:** cookie > `AuthSession.user.theme` > default `"demacia"`. Cookie wins over auth so a user can override their account theme per-device for unauthenticated browsing.
- **D-03 — Middleware integration model:** sync `tower::Layer` that runs AFTER `auth_layer` (so `AuthSession` is already populated in request extensions) and BEFORE `leptos_routes_with_context`'s sync context closure. The layer reads cookie + `AuthSession` from request extensions, resolves the theme, and inserts a typed `InitialTheme(String)` back into request extensions. The leptos_routes context closure reads it synchronously via the request extensions API. Avoids the async-extractor-in-sync-context problem documented at `main.rs:82-83`.
- **D-04 — Cookie write side:** client-side after `set_user_theme` server fn resolves successfully. The DB persist is the source of truth for authenticated users; the cookie is the per-device override for unauthenticated reloads (and for authenticated users who want a different theme on this device than their account default).
- **D-05 — Theme value validation:** middleware MUST validate the cookie value against `{"demacia", "pandemonium"}` before injecting. Reject invalid values (treat as cookie absent → fall through to auth or default). Mitigates trivial-tampering: a forged cookie can only flip between the two valid regions, not inject XSS or other strings.
- **D-06 — First-paint behavior:** when neither cookie nor `AuthSession.user.theme` is set (e.g., a brand-new visitor's first page load), SSR provides `InitialTheme("demacia")`. The post-hydration ThemeToggle component still does its DB read on mount but is now a no-op when DB matches the SSR value (which it does, because SSR served the same demacia default). Acceptable: a fresh visit gets demacia structurally; if they later toggle, the click writes the cookie AND fires the DB persist, and subsequent reloads SSR-render the chosen region.
- **D-07 — No anti-FOUC script changes:** the existing anti-FOUC script in `<head>` (from Phase 17) reads `data-theme` from the cookie-injected `<html>` attribute, which is what SSR sets. No script changes needed; the script becomes a no-op for the cookie-injection case (the attribute is already correct on initial paint).
- **D-08 — region-diff threshold:** plan-phase will instruct re-baselining as part of the verification step; the threshold update happens AFTER the new baselines land and the actual measured pixel diffs are known. Educated estimate: bump `REGION_DIFF_THRESHOLD` from 0.005 (current; catches token swap) to somewhere in [0.04, 0.08] (catches structural swap, leaves headroom). Final value is empirical, not pre-committed.

## Claude's Discretion

- **Per-test or global cookie injection in Playwright?** The e2e auth fixture (`e2e/tests/fixtures.ts`) currently calls `setRegion` which clicks the toggle. After 18.1 lands, `setRegion` SHOULD set the cookie directly via `page.context().addCookies()` AND THEN reload — otherwise the click flips data-theme but the new region prop is only honored on the NEXT navigation. Plan-phase chooses between (a) update `setRegion` helper, (b) add a new `setRegionAndReload` helper, (c) leave `setRegion` as-is and require tests to reload explicitly. Leaning (a) for backwards compatibility — every existing test calling `setRegion` benefits without code changes.
- **Naming of the new middleware module:** could be inline in `main.rs`, or a new `src/server/theme_layer.rs`. Leaning a new file because it has nontrivial logic (cookie parsing + auth fallback + validation) that benefits from being unit-tested in isolation.
- **Should `set_user_theme` return the new theme?** Currently returns `Result<(), ServerFnError>`. If we make it return `Result<String, ServerFnError>`, the client can use the server-confirmed value to write the cookie (defense-in-depth against the client trusting its own optimistic value). Trade-off: tiny API churn vs slightly stronger correctness. Leaning yes if it's a one-line change.

## Risk surface

- **Auth ordering:** `tower::Layer` ordering matters. The new theme layer MUST run after `auth_layer` (which populates `AuthSession`) but before the leptos_routes context closure. Get the `.layer()` ordering wrong and the layer can't read `AuthSession`, falling back to cookie or default — silently rendering the wrong region for authenticated users with no cookie. Plan-phase's verification step must explicitly test: (a) authenticated user with no cookie sees their account theme; (b) authenticated user with a cookie sees the cookie value (not the account theme); (c) unauthenticated user with no cookie sees demacia; (d) unauthenticated user with cookie sees the cookie value.
- **First-toggle race:** click PANDEMONIUM → optimistic DOM data-theme="pandemonium" + spawn_local set_user_theme + cookie write. If user navigates BEFORE the cookie lands (very fast click → nav), the next SSR request reads no cookie, no DB (assuming unauthenticated) → renders demacia, and the page hard-reloads showing the wrong region for one frame. Mitigation: write the cookie synchronously in the same JS turn as the data-theme attribute update (before the await for the server fn), so even sub-100ms navigations honor it.
- **Cookie injection in Playwright:** `setRegion` helper currently clicks the toggle. After 18.1, structural region rendering depends on cookie + reload. Tests that call `setRegion` then `goto()` will work; tests that call `setRegion` and check the CURRENT page's structure will see the old structure until reload. Plan-phase mitigates by updating the helper to set the cookie + reload.
- **REGION_DIFF_THRESHOLD bump:** the existing 0.005 threshold is now too low — every scoped route will produce 5–15% pixel diff post-fix. If we don't update the threshold during 18.1, region-diff.spec.ts will start failing for the OPPOSITE reason: the diff will exceed an upper bound we never had (the test only asserts `>` not `<`). Lower the threshold to ~0.04 to keep catching "regions actually differ" while leaving headroom. Plan-phase orchestrates the re-baseline + threshold bump as one task.

## Verification approach

The verifier for 18.1 MUST do a runtime browser sweep, not just source-code grep. Failing to do this is exactly the W-PRE-10 failure mode that caused the bug to ship in Phase 18. The verification step in the plan should:

1. Start dev server (`cargo leptos watch`), wait for ready
2. agent-browser registers a fresh user, lands on /solo
3. Toggle to PANDEMONIUM, reload (cookie now set)
4. Visit all 11 scoped pages in turn; on each, run `agent-browser snapshot` and grep for Pandemonium-only markers from the per-page content contracts (RiotTape, Glitch eyebrow text, zine card classes, etc.)
5. Repeat for Demacia (toggle back, reload, sweep)
6. Spot-check 3 utility routes (login, profile, opponents) — confirm IDENTICAL rendering across regions (REQ-7 holds)
7. Re-run `region-diff.spec.ts` with the new threshold; confirm passes
8. Re-baseline all 24 scoped baselines (delete + re-snapshot + commit)
9. Mark UI-18-RUNTIME-01 as resolved in 18-HUMAN-UAT.md
