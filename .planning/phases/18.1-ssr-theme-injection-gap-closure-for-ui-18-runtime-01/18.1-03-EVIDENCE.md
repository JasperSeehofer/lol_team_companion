## Investigation Flags

### Flag 1 (NEW, surfaced by this sweep): WASM hydration panic on every Pandemonium page

When a Pandemonium-themed page loads in a browser (e.g. /solo with `lol_companion_theme=pandemonium`), the JS console captures:

```
[error] panicked at /home/jasper/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/tachys-0.2.14/src/html/mod.rs:217:14:
called `Option::unwrap()` on a `None` value
```

…with a full backtrace through `console_error_panic_hook` / Leptos's `tachys` hydration code. The WASM runtime crashes during the hydrate-from-SSR phase, leaving the page in a "SSR-rendered HTML only" state — no event handlers fire (clicks on mode-toggle buttons, theme-toggle buttons, etc. do nothing). Demacia pages hydrate cleanly with no panic; the panic is specific to Pandemonium SSR HTML — the structural Pandemonium branch in some Leptos primitive produces a markup tree that the client-side hydration code cannot reconcile against the WASM render of the same component.

**Why this didn't surface in Phase 18:** Pre-18.1, the SSR ALWAYS rendered the Demacia branch (the root cause of UI-18-RUNTIME-01). Hydration matched fine because both SSR and the client's first WASM render produced the same Demacia tree. Phase 18.1's middleware now makes the SSR render the Pandemonium tree when the cookie says so, exposing the latent SSR/hydrate mismatch for the first time.

**Why it doesn't block the W-PRE-10 / UI-18-RUNTIME-01 gap-closure question:** UI-18-RUNTIME-01 was specifically: "Does the SSR-rendered DOM contain the Pandemonium structural branches?" The answer is now empirically YES across all 7 scoped routes × 28 markers. Hydration failure is a **separate, newly-surfaced bug** that should be tracked as its own finding and fixed in a follow-up phase (recommend Phase 18.2 — "Pandemonium Hydration Reconciliation"). Hydration failure does NOT cause the page to render incorrectly — it just disables client-side interactivity (mode toggles, draft slot drag-drop, etc). The SSR-rendered Pandemonium structural branches are still visible and visually correct.

**Reproduction:** Open any /solo, /team/dashboard, /draft, etc. with `lol_companion_theme=pandemonium` set, observe the console after page load. The Demacia equivalent (cookie=demacia) produces no panic.

**Recommended disposition for THIS plan:** Note the flag in this evidence file and in 18.1-03-SUMMARY.md; do NOT block UI-18-RUNTIME-01 resolution on it; do open a new finding (UI-18.1-HYDRATE-01) in 18-HUMAN-UAT.md or roadmap. The user is the final arbiter; the human-verify checkpoint task in this plan presents the choice.

### Flag 2 (informational): Phase 18-08 mode toggle is DB-persisted, NOT SSR-rendered for non-default modes

As noted in the Mode-toggle SSR defaults section above, the mode-toggle on /draft, /solo, /team/dashboard always SSR-renders the region-coupled default (resolve_mode("auto", region)). A user's explicit DB-persisted mode pick only takes effect after WASM hydration + auth_user Resource fetch + Effect run on the client. This is **by design per Phase 18-08** — the resource fetch is async and Leptos SSR Effects are no-ops. But combined with Flag 1 above (hydration panic on Pandemonium), this means: **a Pandemonium-cookie user who has persisted `solo_mode=constellation` to their account will SSR-load `/solo` in forge mode and never reach constellation because hydration crashes before the Effect runs.**

Track in 18.1-04 or 18.2 alongside Flag 1.
