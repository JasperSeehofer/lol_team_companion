## Spacing Scale

Source: established in Phases 12–14, carried forward unchanged. Two additional named tokens added to cover inherited values from Phase 12 GoalPlaceholders and Phase 13 scoreboard.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap-1`), badge padding, graph axis label padding |
| sm | 8px | Compact element spacing (`gap-2`, `px-2 py-1`), table cell vertical padding |
| md | 16px | Default element spacing (`px-4`, `gap-4`), card inner padding |
| lg | 24px | Section padding (`py-6`, `gap-6`), card grid gap |
| xl | 32px | Layout gap between major stacked sections on solo dashboard |
| 2xl | 48px | Major section breaks, page top/bottom padding |
| 3xl | 64px | Page-level spacing (not used in dense data views) |
| card-grid-gap | 12px (`gap-3`) | Goal card grid (3-up); matches Phase 12 GoalPlaceholders + StatCard grid |
| table-row-md | 44px (`h-11`) | Champion trends table row height; matches Phase 13 scoreboard row height |

Exceptions:
- LP graph SVG height: 160px — fixed height regardless of viewport width; full container width (`w-full`)
- LP graph y-axis label column: 48px wide, right-aligned text-xs
- LP graph x-axis label row: 24px tall below the SVG plot area
- Goal card inner padding: `p-4` (16px) — matches existing GoalPlaceholders anatomy
- Goal edit form inside card: `pt-3 mt-3 border-t border-divider` to separate edit fields from card header without a modal
- Time-window toggle button: minimum 32px tall — desktop-first; pill-style
- Progress bar height: `h-2` (8px) — slender, subordinate to the value readout

## Color

Source: `input.css` semantic tokens. All classes listed here, never raw hex or oklch.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-base` | Page background |
| Secondary (30%) | `bg-surface` / `bg-elevated` | LP graph container card, goal cards, trends table card |
| Accent (10%) | `bg-accent` / `text-accent` | See reserved list below |
| Destructive | `text-red-400` / `bg-red-500/10` | Error banners only (ErrorBanner component) |

**Accent reserved for:**
1. LP graph line stroke — `stroke-accent` (the primary data series line; makes it immediately identifiable on the dark card surface)
2. Time-window toggle active state — `bg-accent text-accent-contrast` (active pill)
3. "Set Goal" CTA button on unset goal cards — `bg-accent hover:bg-accent-hover text-accent-contrast font-semibold` (primary action)
4. "Save Goal" button in goal inline edit form — same accent CTA style
5. Inline edit text links (Edit Goal button on each goal card) — `text-accent`
6. Filter toggle text links (champion trends min-games toggle) — `text-accent`

**NOT accent:** Goal achieved / on-track status badge fill — use `bg-emerald-500/20 text-emerald-400 border border-emerald-500/30` (matches win badge pattern from Phases 13–14). LP graph axis grid lines, goal "behind" progress bars (use `bg-accent/30` dimmed), champion trends sort arrow indicators, "Show all" toggle.

Additional semantic colors:
- LP graph axis grid lines: `stroke-border-divider opacity-50` (subtle horizontal grid lines)
- LP graph data points (dots at each snapshot): `fill-accent stroke-accent`
- LP graph area fill below line: `fill-accent/10` (very light fill between line and x-axis)
- Goal progress bar track: `bg-elevated` (matches Phase 13 PerformanceBarChart bar track)
- Goal progress bar fill (in-progress): `bg-accent/50` (dimmed accent, not full — communicates partial progress)
- Goal progress bar fill (achieved/on-track): `bg-emerald-500/60` (green — matches win badge convention)
- Rank Target LP-to-go text: `text-muted` — the "653 LP to go" line reads as informational, not urgent
- CS/min and Deaths on-track badge: `bg-emerald-500/20 text-emerald-400 border border-emerald-500/30`
- Deaths behind target: progress bar fill stays `bg-accent/50`; no red coloring — deaths goal is directional guidance, not an alarm
- Champion trends win rate: no color tinting on the number; column header only
- Champion trends min-games filter toggle: `text-accent text-sm` link style for "Show all champions" / "Hide low sample"
- "Need N more games" state: plain `text-muted text-sm` — no color treatment

### New: LpHistoryGraph
**Location:** `src/pages/solo_dashboard.rs` — inline component (no separate file unless >150 LOC)
**Placement on page:** new section between `RankedBadgeSection` and `MatchListSection` (D-06)
**Data source:** `get_lp_history(user_id, window)` server fn returning `Vec<RankedSnapshot>` sorted by `snapshotted_at`

**Anatomy:**

```
┌──────────────────────────────────────────────────────────┐  bg-elevated border border-divider rounded-xl p-4
│  LP History                    [7d] [30d] [90d] [All]   │  section header + time-window toggle inline
├─────────────┬────────────────────────────────────────────┤
│  Challenger │                                            │  y-axis label column: 48px wide, text-xs text-muted
│  Master     │                        ●                  │  tier boundary labels at computed rank score values
│  Diamond    │              ●────────╯                   │  SVG line: stroke-accent stroke-width="2"
│  Platinum   │    ●─────────╯                            │  data points: fill-accent r="3"
│  Gold       │───╯                                       │  area fill: fill-accent/10
│  Silver     │                                           │
├─────────────┴────────────────────────────────────────────┤
│             Jan 10        Jan 20       Jan 30            │  x-axis date labels: text-xs text-muted
└──────────────────────────────────────────────────────────┘
```

**Dimensions:** SVG viewBox height 160px; full container width; responsive (100% of card content area). Y-axis label column is part of the SVG, not a separate element.

**Y-axis scale:** cumulative rank score (Iron 4 = 0, +100/division, +400/tier; Master+ = 2800 + raw LP). Y-axis shows 4–5 tier boundary labels (adapt to range visible in data — always show the current tier and 2 above/below). No raw numbers; tier names only (e.g. "Gold", "Platinum").

**Tier emblem labels on y-axis:** text labels only (tier name string) — no image emblem on the axis to keep SVG simple. The ranked badge section already shows the tier emblem image. Source: tier name from `RankedSnapshot.tier` — no Data Dragon image fetch needed for the axis.

**Time-window toggle pills:**

```
[7d]  [30d]  [90d]  [All-time]
```

- Active pill: `bg-accent text-accent-contrast text-xs px-3 py-1.5 rounded-full font-semibold`
- Inactive pill: `bg-elevated border border-divider text-muted text-xs px-3 py-1.5 rounded-full hover:border-outline hover:text-secondary transition-colors`
- Toggle state: `RwSignal<LpWindow>` (enum: `Days7 | Days30 | Days90 | AllTime`) — default `Days30` (D-02)
- Toggle changes trigger `lp_history_resource.refetch()` — no client-side filter; server fn takes window as param

**Hover tooltip:** `<div>` overlay positioned via absolute CSS at cursor location. Only visible in `#[cfg(feature = "hydrate")]` context. Content: tier/division/LP on first line (`text-sm text-primary font-semibold`), snapshot date on second line (`text-xs text-muted`). Style: `absolute bg-elevated border border-divider rounded-lg px-3 py-2 text-sm shadow-lg pointer-events-none z-10 min-w-32`. Hover tracking uses `on:mousemove` on the SVG element to update `RwSignal<Option<(f64, f64, RankedSnapshot)>>`. The data point nearest cursor (within 12px) triggers tooltip display.

**Single data point:** render a centered dot only (no line). Graph still renders; dot sits centered horizontally.

**Empty state (no snapshots):** Replaces the SVG entirely — see Copywriting Contract. Shows `EmptyState` component with Sync button.

**Skeleton loading:** `SkeletonCard height="h-48"` — covers graph + header row height.

**Progress bar spec:**
- Container: `w-full bg-elevated rounded-full h-2 mt-2`
- Fill: `h-2 rounded-full transition-all` with `style="width: {pct}%"`
- Fill color (in-progress): `bg-accent/50`
- Fill color (achieved): `bg-emerald-500/60`
- Fill capped at 100% visually; never overflows

**Progress calculation:**
- Rank Target: `(current_rank_score / target_rank_score) * 100` — LP scale from D-03
- CS/min: `(current_avg / target) * 100`, capped at 100%
- Deaths/game: `(target / current_avg) * 100` (inverted — lower deaths = more progress), capped at 100% when `current_avg <= target`

**Status badges:**
- Rank Target achieved: "Achieved" (`bg-emerald-500/20 text-emerald-400 border border-emerald-500/30 text-xs font-semibold rounded-full px-2 py-0.5`)
- CS/min on-track: "On track" (same badge style)
- Deaths on-track: "On track" (same badge style)
- In-progress: no badge — progress bar communicates state visually

**Edit Goal button:** `text-accent hover:text-accent-hover text-xs transition-colors` — text link style, bottom-right of card

**Edit state (inline, no modal):**

```
┌──────────────────────────────────┐
│  [icon]  Rank Target             │
│  ─────────────────────────────   │  border-t border-divider mt-3 pt-3
│  Tier:  [Gold        ▾]          │  <select> styled with existing dropdown classes
│  Div:   [II          ▾]          │  (hidden for Master+)
│                                  │
│  [Save Goal]   [Discard]         │  Save Goal: bg-accent text-accent-contrast text-xs px-3 py-1.5 rounded-lg
└──────────────────────────────────┘     Discard: text-muted hover:text-secondary text-xs
```

**Edit form details:**
- Rank Target: two `<select>` elements — Tier (Iron/Bronze/Silver/Gold/Platinum/Emerald/Diamond/Master/Grandmaster/Challenger) and Division (I/II/III/IV — hidden when tier is Master/Grandmaster/Challenger)
- CS/min: `<input type="number" min="0" max="15" step="0.1">` — range enforced by `min`/`max` attributes; client-side guard rejects outside 0–15 before server fn call
- Deaths/game: `<input type="number" min="0" max="20" step="1">` — range 0–20
- Input style: `bg-surface/50 border border-outline/50 rounded-lg px-3 py-2 text-primary text-sm focus:outline-none focus:border-accent/50 transition-colors w-full` — matches existing form input pattern (from Phase 14)
- Form labels: `text-xs text-muted uppercase tracking-wider mb-1`
- Edit state stored in `RwSignal<bool>` per card (`editing_rank`, `editing_cs`, `editing_deaths`)
- Save Goal calls `upsert_personal_goal` server fn; `StatusMessage` shown inline below Save Goal/Discard row on error; card collapses back to active state on success and `goal_progress_resource.refetch()` is called

**Goal card component extraction:** If any single goal card exceeds 100 LOC in the view macro, extract to a named sub-component (`RankTargetCard`, `CsGoalCard`, `DeathsGoalCard`) within `solo_dashboard.rs`. Do not create separate files.

## Interaction Contracts

### LP Graph hover

- SVG `on:mousemove` handler (WASM-only, `#[cfg(feature = "hydrate")]`): compute nearest data point within 12px Euclidean distance from cursor; update `RwSignal<Option<TooltipState>>`
- `TooltipState` carries: pixel position (x, y), and snapshot data (tier, division, lp, snapshotted_at)
- Tooltip `<div>` positioned with `style="left: {x}px; top: {y}px"` relative to a `position: relative` wrapper around the SVG
- SVG `on:mouseleave`: clears tooltip signal
- Click on SVG: no action (D-04)

### Goal card edit flow

1. User clicks "Edit Goal" text link
2. `editing_{type}: RwSignal<bool>` → `true`; card body swaps to inline edit form via `{move || if editing.get() { ... } else { ... }}` with `.into_any()` on each branch (leptos-patterns rule 19)
3. User modifies fields; client-side validation on "Save Goal" click:
   - CS/min: reject if parsed value outside 0.0–15.0; show `StatusMessage` error inline
   - Deaths: reject if parsed value outside 0–20; show `StatusMessage` error inline
   - Rank Target: always valid (controlled select values)
4. On valid save: `spawn_local` calls `upsert_personal_goal` server fn; on success, `editing.set(false)` and `goal_progress_resource.refetch()`
5. Discard: `editing.set(false)` with no save — input values discarded (signals reset to current goal value on next render)

### Champion trends sort

- Click any column header → if already sorted by this column, reverse order; otherwise set new sort column, direction=descending
- Sort is client-side: derive a sorted + filtered `Vec<ChampionTrend>` from `Memo<Vec<ChampionTrend>>` combining `sort_col`, `sort_dir`, and `show_all_champions` signals
- No server round-trip on sort change

### Time-window toggles

- LP graph window and champion trends window are independent signals — changing one does not affect the other (D-17)
- LP graph window change: calls `lp_history_resource.refetch()` — server-side filtered
- Champion trends window change: calls `champion_trends_resource.refetch()` — server-side filtered
- Neither window persists to DB; resets to 30d on page load (D-18 discretion — simpler state)

## Implementation Notes for Executor

1. **LP graph SVG coordinate mapping:** Compute `x_scale = (svg_width - y_axis_width) / max(n_points - 1, 1)` for evenly-spaced x positions. Compute `y_scale = (svg_height - x_axis_height) / (max_rank_score - min_rank_score)`. Data point at index `i` maps to `x = y_axis_width + i * x_scale`, `y = svg_height - x_axis_height - (snapshot.rank_score - min_rank_score) * y_scale`. Use `viewBox="0 0 {width} {height}"` with `preserveAspectRatio="none"`.

2. **LP graph area fill:** `<path d="M {first_x},{first_y} L {all_points...} L {last_x},{bottom} L {first_x},{bottom} Z" fill="..." />` rendered behind the line and dots.

3. **SVG semantic-token colors:** Tailwind CSS v4 generates CSS custom properties from `@theme`; SVG `fill` and `stroke` cannot use Tailwind class names directly. Use `class="..."` with `fill-current text-accent` for SVG elements where possible, or inline `style="stroke: var(--color-accent)"`. Check which approach the existing `tree_graph.rs` SVG uses and match it.

4. **Cumulative rank score formula (D-03):** `fn rank_score(tier: &str, division: &str, lp: i32) -> i32`. Tier order: Iron=0, Bronze=1, Silver=2, Gold=3, Platinum=4, Emerald=5, Diamond=6, Master=7, Grandmaster=7, Challenger=7. Division order: IV=0, III=1, II=2, I=3. Score = tier_index * 400 + division_index * 100 + lp. Master+: 2800 + lp (division ignored). Implement in models or db.rs where `RankedSnapshot` conversion happens.

5. **Goal progress resource:** `compute_goal_progress` should return a single struct `GoalProgressPayload { rank: Option<GoalProgress>, cs: Option<GoalProgress>, deaths: Option<GoalProgress>, current_rank: Option<RankedSnapshot> }` so the entire Goals section loads from one server fn call, not three. This keeps the Suspense boundary simple.

6. **`Ok(Vec::new())` for empty trends:** When the user has no ranked games in the selected window, `get_champion_trends` returns `Ok(Vec::new())`. The empty state renders from the empty vec — do NOT return `Err` for absence of data (leptos-patterns rule 44).

7. **Goal edit input reset:** When the user clicks Discard, the input value must revert. Store edit state in local `RwSignal<String>` initialised from the current goal target on card mount; Discard discards the signal change by re-initialising from the resource value on next render (resource refetch is NOT needed — the goal value on the server did not change).

8. **Champion icon fallback:** Use `on:error` on the `<img>` element to swap src to a placeholder. Leptos: `on:error=move |ev| { /* set signal to fallback */ }`. Fallback: `<div class="w-5 h-5 rounded bg-elevated border border-divider/30" />`. Use `into_any()` for the conditional render.

9. **Deaths goal progress bar inversion:** The progress bar for Deaths/game uses inverted logic. A user with 5.2 deaths targeting 4.0 deaths has `pct = (4.0 / 5.2) * 100 = 76.9%`. When deaths ≤ target, `pct = 100%` and bar color switches to `bg-emerald-500/60`. This means "more bar = more progress toward goal" consistently across all three card types.

10. **`stable <For>` key for goal cards:** Use the goal type string (`"rank_target"`, `"cs_per_min"`, `"deaths_per_game"`) as the `key` prop if rendered with `<For>`. Alternatively, the three cards are distinct enough to render as explicit `if/match` branches without `<For>`.

11. **Trends table client-side sort:** Compute `sort_key` closure on `ChampionTrend` matching the active `SortColumn`. Sort in the `move ||` closure inside the table body, operating on the resource data. Use `Memo` if sort recomputes are expensive. The filtered + sorted vec is a pure function of (resource data, sort_col, sort_dir, show_all) — no side effects.
