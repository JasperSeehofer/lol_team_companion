## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server (`do_sync` handler → `sync_solo_matches`) | WASM-side click triggers an authenticated server fn. Already mitigated by axum-login session; this plan does not alter the boundary. |
| Component prop boundary (`SoloDashboardPage` → `LpHistoryGraph`) | New: a `Resource` handle and `RwSignal` cross from parent to child. Both are `Copy` reference handles, not data; no integrity risk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | `lp_history_resource` prop in child | accept | Resource handles are Copy reference types in Leptos 0.8; child cannot mutate the data, only read via `.get()`. No injection vector. |
| T-16-02 | Denial of Service | Three `.refetch()` calls per sync click | mitigate | Existing `syncing: RwSignal<bool>` guard at lines 228 + 260 + `auto_synced` one-shot guard at line 234 already prevent rapid-fire sync triggers. The three refetches add ~3 round-trips per sync (vs. previous 1) — acceptable since sync is user-initiated and rate-limited by the upstream Riot API throttle. |
| T-16-03 | Tampering | Stale-capture in delayed closure | mitigate | Per WASM Rule 56 and Leptos Rule 54: the new `.refetch()` calls run inside `spawn_local`'s `Ok(_)` branch — not inside a `Closure::once` timer — so there is no delayed-capture risk. Verified by reading lines 237-254 (no `set_timeout` between `Ok(n)` and the refetch calls). |
| T-16-04 | Information Disclosure | Cross-user data leak via hoisted resource | accept | `get_lp_history` (existing server fn) authenticates via session cookie and scopes its query to `auth.user.id`. Hoisting changes only the call site, not the auth boundary. The resource still issues one request per `lp_window.get()` change, scoped to the current logged-in user. |
| T-16-05 | Denial of Service | Auto-sync infinite loop after refetch triggers Effect | mitigate | The `auto_synced.get_untracked()` guard at line 234 sets `auto_synced.set(true)` before sync runs (line 235); after refetch fires, the Effect re-evaluates but `!auto_synced.get_untracked()` is now `false`, short-circuiting. D-04 explicitly preserves this guard. |
