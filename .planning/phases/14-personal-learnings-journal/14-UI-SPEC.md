## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, tag chip inline padding vertical, badge padding |
| sm | 8px | Compact element spacing, tag chip horizontal padding, card gap between meta items |
| md | 16px | Default element spacing, card internal padding, form field spacing |
| lg | 24px | Section padding, card grid gap |
| xl | 32px | Layout gaps between major sections |
| 2xl | 48px | Page top/bottom padding |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Filter bar: `py-6` (24px) top/bottom with `px-6` (24px) sides to match page container (source: existing stats.rs pattern)
- Card grid gap: `gap-6` (24px) between cards
- Inline-expanded card section padding: `pt-4` (16px) top border gap inside expansion

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Page heading | 30px (text-3xl) | 700 (font-bold) | 1.2 |
| Card title | 14px (text-sm) | 500 (font-medium) | 1.4 |
| Body / label | 14px (text-sm) | 400 (font-normal) | 1.5 |
| Caption / meta | 12px (text-xs) | 400 (font-normal) | 1.4 |

Source: Established in codebase — `stats.rs` uses `text-3xl font-bold` for page headings; `text-sm` at weight 400/500 for body and labels throughout; `text-xs` for badges, timestamps, and meta. Maximum 4 sizes, 2 weights (normal + medium/bold pairing per element role).

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-base` (gray-950 dark / gray-50 light) | Page background, outer shell |
| Secondary (30%) | `bg-surface` / `bg-elevated` (gray-900 / gray-800) | Cards, filter bar background, form panels |
| Accent (10%) | `bg-accent` (yellow-400 dark default) | Active tab indicator, primary CTA button, filter active state |
| Destructive | `bg-red-700 text-white` / `text-red-400` | Delete learning button only |

Accent reserved for:
- "Save Learning" primary CTA button (`bg-accent text-accent-contrast`)
- Active type/tag filter chips (selected state)
- "Add Learning" CTA button on match detail scoreboard and timeline event panel

Win/Loss badges use semantic status colors, not the accent:
- Win: `bg-emerald-500/20 text-emerald-400 border border-emerald-500/30`
- Loss: `bg-red-500/10 text-red-400 border border-red-500/30`

Type badges (General / Champion / Matchup) use overlay, not accent:
- `bg-overlay text-muted text-xs font-medium rounded px-2 py-0.5`

## Interaction Contracts

### Learning Card (D-08, D-11)

Each card in the grid:

```
┌──────────────────────────────────────────────────┐
│  [Type badge]  [Win badge / Loss badge]  [date]  │
│  Card title (font-medium text-primary, truncate) │
│  Champion icon 20px (if type=Champion/Matchup)   │
│  Section preview: first 80 chars of "What        │
│  happened", text-muted text-xs                   │
│  ─────────────────────────────────────────────   │
│  Tags as chips: [Laning] [Vision] …              │
└──────────────────────────────────────────────────┘
```

- Default state: `bg-elevated/30 border border-divider/30 rounded-xl p-4 hover:bg-overlay/30 transition-all cursor-pointer`
- Expanded state: `bg-accent/10 border border-accent/30 rounded-xl p-4 transition-all`
- On click: card expands inline (no page navigation) — reveals full three sections, edit button, delete button
- Expansion shows full text for all three sections under labeled headings
- Edit opens `/personal-learnings/new?edit={id}` (or switches to inline edit mode at discretion — see note)
- Delete button: `text-red-400/50 hover:text-red-400 text-xs transition-colors` — no confirmation modal; immediate delete with toast undo (matches post_game.rs pattern)

**Card grid breakpoints:**
- Default (< 1024px): 2 columns — `grid grid-cols-2 gap-6`
- lg and above (>= 1024px): 3 columns — `lg:grid-cols-3`

### Filter Bar (D-09, D-10)

Horizontal row of `<select>` dropdowns:

```
[Type: All ▾]  [Champion: All ▾]  [Tag: All ▾]  |  [Sort: Newest first ▾]
```

- Dropdown style: `bg-surface/50 border border-outline/50 rounded-lg px-3 py-1.5 text-primary text-sm focus:outline-none focus:border-accent/50` (matches stats.rs queue filter)
- Filters apply instantly on change (no submit button) — client-side signal filtering
- Sort control placed right-aligned in the same bar row
- Sort by champion renders alpha-grouped list with sticky letter headers (`text-xs text-dimmed uppercase tracking-wider py-2`)

### Learning Form (D-01 through D-05, D-06, D-07)

**Type selector (first decision, tab-style):**
- Three buttons: General | Champion | Matchup
- Active: `px-4 py-2 rounded-lg text-sm font-medium bg-accent text-accent-contrast`
- Inactive: `px-4 py-2 rounded-lg text-sm font-medium bg-elevated border border-divider text-secondary hover:text-primary hover:border-accent transition-colors`

**Conditional field visibility:**
- General: no champion or opponent field shown
- Champion: show champion autocomplete only
- Matchup: show both champion autocomplete and opponent autocomplete

**Three required text areas:**
- Labels: "What happened", "What I learned", "Next time I will..."
- Each: `w-full bg-surface/50 border border-outline/50 rounded-lg px-3 py-2 text-primary text-sm placeholder-dimmed focus:outline-none focus:border-accent/50 transition-colors` with `rows="4"`
- All three required — save blocked if any empty; inline validation message below empty field: `text-red-400 text-xs mt-1`

**Tag multi-select chips:**
Predefined tags rendered as toggle chips:

| Tag label | Maps from event type |
|-----------|---------------------|
| Laning | — |
| Teamfighting | Champion kill event |
| Macro / Rotations | — |
| Vision | — |
| Trading | Champion kill event (secondary) |
| Wave Management | — |
| Objective Control | Dragon kill, Baron kill, Tower kill |
| Mental / Tilt | — |

Selected chip: `rounded-full px-3 py-1 text-xs font-medium bg-accent text-accent-contrast cursor-pointer`
Unselected chip: `rounded-full px-3 py-1 text-xs font-medium bg-overlay text-muted hover:bg-overlay-strong hover:text-secondary transition-colors cursor-pointer`

**Title field:**
- Auto-populated: "{Champion} vs {Opponent} — {Month Day}" for Matchup; "{Champion} — {Month Day}" for Champion; "General — {Month Day}" for General
- Editable `<input>` rendered below tags
- Style: `w-full bg-surface/50 border border-outline/50 rounded-lg px-3 py-2 text-primary text-sm focus:outline-none focus:border-accent/50`

**Pre-fill from query params (D-12, D-13):**
- `?champion=` pre-selects type as Matchup (if both champion+opponent present) or Champion, populates champion field
- `?opponent=` populates opponent field
- `?match_id=` attached silently to the record (not shown as a visible field)
- `?result=win|loss` silently attached
- `?event_ts=` and `?event_name=` rendered as a read-only context banner above the form: `bg-elevated border border-divider rounded-lg p-3 text-sm text-secondary` showing "From match: {event_name} at {event_ts}"
- Auto-suggested tag chip highlighted (pre-selected) based on event-type mapping above

**Save / Cancel row:**
- Save: `bg-accent hover:bg-accent-hover text-accent-contrast font-semibold px-5 py-2.5 rounded-lg text-sm transition-colors` — label "Save Learning"
- Cancel: `bg-elevated border border-divider text-secondary hover:text-primary px-5 py-2.5 rounded-lg text-sm transition-colors` — navigates back to `/personal-learnings`

## Implementation Notes for Executor

1. Route `/personal-learnings` is protected — apply the same auth guard pattern used on all protected pages (`get_current_user()` + client-side redirect to `/auth/login` if None).

2. Card inline expand: use a `RwSignal<Option<String>>` holding the currently expanded learning ID. Toggle on card click. No animation required beyond the class swap (transition-all handles it).

3. Filter logic is client-side — load all learnings once in a `Resource`, then derive a filtered+sorted `Memo` or compute inside a `move ||` closure. No server round-trip per filter change.

4. Sort by champion: collect into a `BTreeMap<String, Vec<LearningCard>>` keyed by champion name; render as sections with sticky letter headers. General learnings group under "General" at top.

5. "Add Learning" from match detail passes data only as URL query params — the learning form page reads them via `use_query_map()` on load and pre-populates signals. No server state shared between pages.

6. `personal_learning` table is user-scoped, not team-scoped. Server functions must filter by `user_id`, not `team_id`. Return `Ok(Vec::new())` (not `Err`) when no entries exist.

7. Debounced auto-save is NOT used on this form — learnings are drafts the user explicitly submits. Only explicit "Save Learning" click triggers the server function.
