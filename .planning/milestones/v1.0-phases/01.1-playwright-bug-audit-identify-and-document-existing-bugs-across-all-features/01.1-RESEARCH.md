# Phase 01.1: Playwright Bug Audit — Research

**Researched:** 2026-03-14
**Domain:** Playwright e2e testing · Leptos 0.8 WASM reactive patterns · SurrealDB draft/tree data flows
**Confidence:** HIGH

## Summary

This phase inserts a systematic Playwright audit before any new feature work. Two bugs are already confirmed by the user: (1) saving a draft does not show it in the saved drafts list, and (2) switching between drafts in the tree drafter makes the app unresponsive. The goal is to discover all bugs across all 14 routes — not to fix them — and produce a structured bug report.

The existing e2e suite covers page-load smoke tests (`pages.spec.ts`), auth flows (`auth.spec.ts`), and the draft→game-plan→post-game pipeline (`pipeline.spec.ts`). Coverage gaps are significant: no CRUD mutation tests, no champion pool interactions, no team management flows, no tree-drafter node editing, no stats, no opponents/action-items pages, and no tests for the "save draft then list appears" scenario that is already known to be broken.

The audit approach is: write new Playwright tests for every meaningful user flow, run them against the live dev server, and record each failure as a structured bug. Tests written during the audit will persist as regression coverage for later fix phases.

**Primary recommendation:** Write bug-audit tests feature-by-feature, using the existing `authedPage` fixture with a team pre-created inside each test that needs one, and record all failures in a `BUG-REPORT.md` in the phase directory.

### Pattern 1: Fixture Extension for Team Setup

Most features require a team. The fixture should be extended with a `teamPage` variant that pre-creates a team:

```typescript
// Source: derived from existing fixtures.ts pattern
export const test = base.extend<Fixtures>({
  authedPage: async ({ page }, use) => {
    await authenticatePage(page);
    await use(page);
  },
  teamPage: async ({ page }, use) => {
    await authenticatePage(page);
    await createTeam(page, `AuditTeam_${Date.now()}`);
    await use(page);
  },
});
```

### Pattern 2: CRUD Test Structure

Each CRUD audit test follows: create → verify in list → update → verify updated → delete → verify gone.

```typescript
test("draft: save then appears in list", async ({ teamPage }) => {
  const page = teamPage;
  await page.goto("/draft");
  await page.waitForLoadState("networkidle");
  await page.waitForTimeout(1000);

  // Create
  await page.getByRole("textbox").first().fill(`AuditDraft_${Date.now()}`);
  await page.locator('button:has-text("Save Draft")').click();
  await page.waitForTimeout(2000);

  // Verify the button switches to "Update Draft" (draft was assigned an ID)
  await expect(page.locator('button:has-text("Update Draft")')).toBeVisible({ timeout: 5000 });

  // Verify name appears in saved drafts list on the same page
  const savedSection = page.locator("text=Saved Drafts").locator("..");
  await expect(savedSection.locator(`text=AuditDraft_`)).toBeVisible({ timeout: 5000 });
});
```

### Pattern 3: Unresponsiveness Detection

For the tree-drafter switching bug, test that the page remains interactive after switching drafts:

```typescript
test("tree-drafter: remains interactive after switching trees", async ({ teamPage }) => {
  // Create two trees, switch between them, verify UI still responds to clicks
  await page.goto("/tree-drafter");
  // ... create tree A, create tree B, click tree A ...
  // Verify a button still responds
  const btn = page.locator('button:has-text("Add Node")');
  await expect(btn).toBeEnabled({ timeout: 3000 });
  await btn.click();
  // Verify click was processed (not frozen)
  await page.waitForTimeout(500);
  // Check the page is still interactive by verifying another element changed state
});
```

### Pattern 4: Bug Documentation

When a test fails, it is promoted to a documented bug. Tests that document known bugs should:
- Use `test.fail()` annotation to mark them as expected failures during the audit
- Include a comment with the suspected root cause

```typescript
test.fail("KNOWN BUG: saved draft does not appear in list immediately", async ({ teamPage }) => {
  // ... reproduce the bug ...
});
```

### Anti-Patterns to Avoid
- **Hardcoded waitForTimeout(5000):** Use `waitForLoadState("networkidle")` + a short `waitForTimeout(500)` for WASM settle. Long arbitrary timeouts mask timing bugs.
- **Testing without a team:** `save_draft` requires a team (returns `ServerFnError("You must be in a team to create a draft")`). Tests that skip team creation will fail with misleading errors.
- **Using `.unwrap()` in test selectors:** Be specific — `page.locator('button:has-text("Save Draft")')` is better than `page.locator("button").first()`.
- **Ignoring console errors in audit tests:** The audit should capture ALL console errors (except known 404s) as potential bug signals.

## Common Pitfalls

### Pitfall 1: "Save Draft" Requires a Team
**What goes wrong:** `save_draft` server fn calls `db::get_user_team_id()` and returns `ServerFnError("You must be in a team to create a draft")` if user has no team. The UI may show "Error: ..." in the status message rather than crashing, making the bug look like a UI bug instead of a prerequisite error.
**Why it happens:** New test users have no team. The draft page does not redirect users without teams.
**How to avoid:** Always call `createTeam()` before any draft-related audit test.
**Warning signs:** `button:has-text("Update Draft")` never appears after clicking "Save Draft".

### Pitfall 2: WASM Effect Settle Delay After Navigation
**What goes wrong:** After `page.goto("/draft")`, Leptos SSR renders HTML, then WASM hydrates, then Effects fire (including URL param auto-load). Asserting too early catches the SSR state, not the hydrated state.
**Why it happens:** Rule 56 in CLAUDE.md — WASM Effect settle requires `waitForTimeout(500)` after `waitForURL`.
**How to avoid:** Pattern: `await page.waitForLoadState("networkidle"); await page.waitForTimeout(500);` after every navigation that loads data.

### Pitfall 3: Server Fn URLs Have Hash Suffixes
**What goes wrong:** Trying to call server fns directly via `page.request.post("/api/save_draft")` will fail — the real URL is `/api/save_draft9651846416477648343`.
**Why it happens:** Leptos `#[server]` macro appends a hash to prevent conflicts.
**How to avoid:** Test server fns indirectly through the UI, not via direct API calls in audit tests. (Direct API testing is already covered by `scripts/api_test.sh` and `flow_test.sh`.)

### Pitfall 4: The Known "Saved Drafts List" Bug
**What goes wrong:** After `save_draft` succeeds (`Ok(id)` returned, `set_loaded_draft_id` updated, button switches to "Update Draft"), `drafts.refetch()` is called. However, the refetch fires `list_drafts()` which queries by team_id — and the newly saved draft appears in that query. The bug may be that `list_drafts` on the page renders inside a `Suspense` that re-reads `drafts.get()`, and the `StoredValue` for `champ_url_map` is created inside the reactive closure on every render. This needs browser verification to isolate. Possible causes:
  1. `drafts.refetch()` fires but the `Suspense` component doesn't re-render because the reactive subscription was dropped.
  2. The `list_drafts()` server fn succeeds but the DB query itself races with the just-committed transaction.
  3. The `drafts` Resource is initialized at component mount but `refetch()` fails silently.
**Warning signs:** "Saved!" message appears but draft name does not appear in the list below.

### Pitfall 5: Tree Drafter Unresponsiveness on Draft Switch
**What goes wrong:** Switching between trees (clicking a different tree in the sidebar) sets `suppress_autosave(true)`, calls `clear_editor()`, then sets `selected_tree_id`. The auto-save Effect tracks multiple signals. If the `setTimeout(0)` callback that re-enables `suppress_autosave` fires while a node's signals are being reset, another Effect run may slip through with `suppress_autosave = false` before all signals are updated, triggering an auto-save with inconsistent state. This could cause a save that blocks the UI.
**Why it happens:** Rules 54-55 in CLAUDE.md describe this exact pattern. The `setTimeout(0)` microtask window between `suppress_autosave.set(true)` and `suppress_autosave.set(false)` may not be wide enough if the batch signal updates cause multiple synchronous Effect invocations.
**Warning signs:** After clicking a different tree, no buttons respond to clicks; `node_label` input doesn't update; the page appears loaded but is frozen.

### Pitfall 6: Pages.spec.ts Content Regex is Permissive
**What goes wrong:** `pages.spec.ts` uses regexes like `/draft|ban|pick/i` for `/draft`. If the page shows an error banner containing "draft" but fails to render content, the test still passes.
**Why it happens:** The regex was designed for breadth, not specificity.
**How to avoid:** Audit tests should use more specific locators to verify meaningful content rendered (e.g., the champion picker grid appeared, not just "draft" text anywhere).

```typescript
// Source: derived from existing pipeline.spec.ts pattern (lines 25-40)
async function createTeam(page: Page, name: string): Promise<void> {
  await page.goto("/team/roster");
  await page.waitForLoadState("networkidle");
  await page.waitForTimeout(500);

  const teamNameInput = page.locator('input[name="name"]');
  if (await teamNameInput.isVisible({ timeout: 3000 }).catch(() => false)) {
    await teamNameInput.fill(name);
    const regionSelect = page.locator('select[name="region"]');
    if (await regionSelect.isVisible({ timeout: 1000 }).catch(() => false)) {
      await regionSelect.selectOption("NA");
    }
    await page.click('button:has-text("Create Team")');
    await page.waitForURL("**/team/dashboard", { timeout: 15000 });
    await page.waitForLoadState("networkidle");
    await page.waitForTimeout(1000);
  }
}
```

## Open Questions

1. **Root cause of "saved draft not in list" bug**
   - What we know: `save_draft` returns `Ok(id)`, `set_loaded_draft_id` is set, `drafts.refetch()` is called
   - What's unclear: Whether `list_drafts()` is actually returning the new draft in its response, or whether the Suspense reactive subscription is broken
   - Recommendation: Add a `console.log` in a dev run to capture the refetch response, OR write the audit test first and use Playwright's `page.on("response")` to intercept the server fn call and verify the response body

2. **Scope of tree-drafter unresponsiveness**
   - What we know: Switching between drafts triggers the bug; `suppress_autosave` pattern is in place per rules 54-55
   - What's unclear: Whether it freezes all WASM (panic) or just the tree component (signal loop); whether deleting a tree while one is selected is a separate trigger
   - Recommendation: Audit test should check for WASM panics via `page.on("pageerror")` during the switch; if panic, inspect stack trace

3. **Opponents and action-items pages coverage**
   - What we know: These routes exist (`/opponents`, `/action-items`) and are not in `AUTHED_PAGES` in `pages.spec.ts`
   - What's unclear: Their full feature set; `pages.spec.ts` does not include them
   - Recommendation: Include them in `audit-misc-pages.spec.ts` with basic load + empty-state checks
